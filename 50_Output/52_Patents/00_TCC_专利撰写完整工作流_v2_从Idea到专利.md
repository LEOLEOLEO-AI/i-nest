# TCC 专利撰写完整工作流

> **一句话：** 从 idea 开始 → 检索现有技术 → 评估可专利性 → 撰写交底书 → 构建权利要求 → 红队审查 → 生成申请文件包。
> **核心约束：** 每步产出可追溯的中间文件；legal gate 不可跳过；不伪造发明人/审查意见/代理确认。

---

## 一、流程总图

```
IDEA
 │
 ├─[1. 策略入口]── R1 Strategy Intake ──→ IP投资备忘录（是否值得申请专利？）
 │
 ├─[2. 现有技术检索]── R4 Prior-Art Diligence ──→ 现有技术矩阵 + 区别映射表
 │
 ├─[3. 可专利性评估]── inventive concept distillation ──→ 新颖性/创造性/实用性报告
 │
 ├─[4. 交底书撰写]── R3 Drafting Pipeline ──→ 完整交底书 MD
 │
 ├─[5. 权利要求构建]── R5 Claim Architecture ──→ 权利要求树（独立+从属+多类别）
 │
 ├─[6. 红队审查]── R6 Patent Red Team ──→ 无效风险 + 规避设计 + 侵权可检测性
 │
 ├─[7. 说明书+附图]── Specification Review ──→ 完整说明书 + 正式附图
 │
 └─[8. 申请文件打包]── R7/R10 Filing Package ──→ 递交就绪包（含legal gate）
```

## 二、每步详细操作

### 步骤 1：策略入口（R1 Strategy Intake）

**触发问题：** 这个 idea 值不值得申请专利？

| 项目 | 内容 |
|------|------|
| **输入** | 技术方向、产品路线、竞争情报 |
| **评估维度** | ① 技术壁垒强度 ② 竞品侵权可能性 ③ 商业价值 ④ 标准必要性 ⑤ 申请时机 |
| **输出** | `R1_IP_Memo_[专利名].md` — IP 投资备忘录 |
| **决策** | GO（进入步骤2）/ NO-GO（存档观察）/ HOLD（补充实验数据后再评估） |

**TCC 当前状态：**
- R1（SDI架构）：✅ GO — SDI系统架构是核心壁垒，易被抢位
- R2（网内归约）：✅ GO — 网内浮点归约是关键技术区分点

### 步骤 2：现有技术检索（R4 Prior-Art Diligence）

**触发问题：** 最接近的现有技术是什么？我们到底新在哪里？

| 项目 | 内容 |
|------|------|
| **检索范围** | CNIPA / USPTO / WIPO / EPO / Google Patents / 学术论文 |
| **检索策略** | 关键词 + IPC分类号 + 申请人追踪 + 前引/后引扩展 |
| **输出** | `R4_Prior_Art_Matrix_[专利名].md` |

**输出格式（逐特征对比表）：**

| 对比项 | 现有技术1 | 现有技术2 | ... | 本发明 |
|--------|----------|----------|-----|--------|
| 文献/专利号 | CNxxx / USxxx | ... | — |
| 相关公开内容 | （摘录） | ... | — |
| 匹配的权利要求要素 | （哪些相同） | ... | — |
| **缺失的权利要求要素** | **（哪些不同）** | ... | — |
| **区别技术特征** | **（本发明独有的）** | ... | — |
| 技术效果证据 | （实验数据支撑） | ... | — |
| 组合风险 | 高/中/低 | ... | — |

**硬性门槛：**
- 每个权利要求要素必须在检索矩阵中有对应行
- 现有技术仅作边界证据，不能作为本发明的权利要求支撑
- ⚠️ 不声称"具有专利性/新颖性/创造性"——这些是审查员的结论，不是我们的自评

**TCC 当前状态：**
- R1：⬜ 待执行
- R2：⬜ 待执行（重点关注 SHARP / In-Network Computing 专利族）

### 步骤 3：可专利性评估（Inventive Concept Distillation）

**触发问题：** 本发明的"发明构思"能否用一句话说清？

**发明构思蒸馏公式：**

```
因为 [已知系统的局限性]，
本发明采用 [具体技术手段]，
改变 [可测量的技术状态/控制/结果]，
从而产生 [技术效果]。
```

**拒绝弱表述（weak formulations）：**
- ❌ "用AI优化"
- ❌ "用传感器监测"
- ❌ "远程App控制"
- ❌ "提高精度"（无具体机制）
- ❌ "动态调整"（无输入/规则模型/输出）

**强表述示例（R1 — SDI架构）：**

```
因为 [现有互连网络仅提供点对点数据搬运，不具有原语级语义]，
本发明采用 [在交换节点中集成通信原语核阵列 + 双缓冲原子交换拓扑重构]，
改变 [互连拓扑从设计时固定变为运行时微秒级可编程]，
从而产生 [集合通信操作在网络内部原生执行，端到端延迟降低50-70%]。
```

**强表述示例（R2 — 网内归约）：**

```
因为 [现有网内计算仅支持整数归约，不能满足深度学习FP16/BF16浮点精度需求]，
本发明采用 [浮点归约执行单元（FPRU），内部高精度累加 + 流水线边传边算]，
改变 [梯度归约从"先传输完毕再归约"变为"边传输边归约"]，
从而产生 [AllReduce延迟降低40-60%，且精度与端节点软件归约无统计差异]。
```

**输出：** `R3_Patentability_Assessment_[专利名].md`

**TCC 当前状态：**
- R1：⬜ 待执行
- R2：⬜ 待执行

### 步骤 4：交底书撰写（R3 Drafting Pipeline）

**章节结构（CNIPA标准）：**

```
1. 发明名称
2. 技术领域
3. 背景技术（含现有技术缺陷 — 来自步骤2的检索矩阵）
4. 发明要解决的技术问题（来自步骤3的蒸馏公式）
5. 技术方案（总体架构 + 核心方法流程）
6. 有益效果（量化对比）
7. 附图说明（每张图的组件编号 + 功能描述）
8. 具体实施方式（≥2个实施例，详细到"本领域技术人员可再现"）
```

**检查清单：**
- [ ] 技术问题 ↔ 技术方案 ↔ 技术效果 三段论逻辑闭环
- [ ] 每个技术特征在实施例中有对应支撑
- [ ] 附图组件编号与说明书完全一致
- [ ] 不包含功能性描述代替结构性限定的情况
- [ ] 不包含没有支撑的性能承诺

**TCC 当前状态：**
- R1：✅ 完整交底书 v2 已完成
- R2：✅ 完整交底书 v2 已完成

### 步骤 5：权利要求构建（R5 Claim Architecture）

**构建方法：**

```
独立权利要求（最宽保护范围）
  ├── 从属权利要求 1: 数据源限定
  ├── 从属权利要求 2: 特征提取限定
  ├── 从属权利要求 3: 状态模型限定
  ├── 从属权利要求 4: 控制动作限定
  ├── 从属权利要求 5: 异常处理限定
  ├── 从属权利要求 6: 硬件布置限定
  └── 从属权利要求 7: 训练/校准限定

平行权利要求类别：
  ├── 方法独立权利要求
  ├── 装置/系统独立权利要求
  ├── 控制器独立权利要求
  ├── 存储介质独立权利要求
  └── 设备/产品独立权利要求
```

**检查清单：**
- [ ] 每个要素有说明书支撑
- [ ] 每个从属权利要求实质性地限缩保护范围
- [ ] 无纯"结果导向"的独立权利要求
- [ ] 无无支撑的性能承诺
- [ ] 先行词基础清晰
- [ ] 独立权利要求不包含非必要技术特征（最宽但有效）

**输出：** `R5_Claims_Tree_[专利名].md` + 权利要求区分说明（多件同日申请时）

**TCC 当前状态：**
- R1：✅ 10条草案（2独立+8从属）
- R2：✅ 12条草案（2独立+10从属）
- ⬜ 两件专利的权利要求区分说明

### 步骤 6：红队审查（R6 Patent Red Team）

**四种攻击角色：**

| 角色 | 攻击方向 | 输出 |
|------|---------|------|
| **审查员** | 显而易见性、清晰度、支撑、主题适格 | 驳回风险列表 |
| **无效请求人** | 现有技术组合、公开不充分 | 无效风险列表 |
| **竞争对手** | 规避设计路径 | 规避风险 + 设计建议 |
| **诉讼律师** | 权利要求可证明性、侵权可检测性 | 可执行性评估 |

**每种攻击输出：**
- 攻击理论（Attack Theory）
- 证据（Evidence）
- 严重度：🔴 Critical / 🟠 High / 🟡 Medium / 🟢 Low
- 缓解措施（Mitigation）
- 负责人（Owner）

**普通驳回理由检查：**
- 权利要求是否包含"功能性限定"而非"结构性限定"？
- 独立权利要求是否可以写得更宽（删除非必要特征）？
- 是否存在"means-plus-function"可能被解释为"说明书记载的对应结构"的风险？

**输出：** `R6_Red_Team_Report_[专利名].md`

**TCC 当前状态：**
- R1：⬜ 待执行
- R2：⬜ 待执行

### 步骤 7：说明书+附图（Specification & Drawings）

**说明书完整性检查：**
- [ ] 每个权利要求要素有说明书支撑
- [ ] 每个权利要求要素有实施例支撑
- [ ] 每个权利要求要素有替代实施例（flexibility）
- [ ] 结构类要素有附图标号引用
- [ ] 每个技术特征有技术效果链接
- [ ] 操作类要素有失败/异常处理描述

**附图要求（CNIPA标准）：**
- 黑白线条图
- 组件编号连续（1, 2, 3...）
- 图中无文字标注（仅在说明书中说明）
- 分辨率 ≥ 300 dpi
- 建议工具：Visio / AutoCAD / Python matplotlib / Inkscape

### 步骤 8：申请文件打包（R7/R10 Filing Package）

**组成文件：**

| # | 文件 | 说明 |
|---|------|------|
| 1 | 发明专利请求书 | 申请人/发明人/代理信息 |
| 2 | 说明书 | 技术领域→实施例 |
| 3 | 权利要求书 | 独立+从属 |
| 4 | 说明书附图 | 4-6张黑白线图 |
| 5 | 摘要 + 摘要附图 | 200字以内 |
| 6 | 同日申请声明 | R1+R2同日递交时必需 |
| 7 | 优先权证明文件 | 如有 |

**Legal Gate（法律合规门槛 — 不可跳过）：**
- [ ] 发明人确认（姓名、顺序、贡献确认）
- [ ] 权利归属确认（职务发明/合作/委托/共有）
- [ ] 保密审查确认（发明是否在中国完成、是否计划PCT/外国申请）
- [ ] 递交渠道授权（账号持有者、电子签名权限、代理机构代码）
- [ ] 费用策略（费减资格、缴费人、缴费账户）
- [ ] 版本哈希（源文件 + XML/包哈希）
- [ ] ⚠️ 明确标注：`official_system_touched=false`, `official_submission_performed=false`, `external_lawyer_involved=false`（AI自助路由）

**输出：** `Filing_Package_[专利名]/` 目录

## 五、关键原则

> 1. **每步产出中间文件，存入知识库，不散落在对话记录里。**
> 2. **Legal gate 不可跳过。** 即使材料已由律师审查，仍需验证。
> 3. **不发明数据。** 不伪造发明人、审查意见、代理确认、技术效果、实验数据。
> 4. **"可上传" ≠ "已递交"。** 明确区分"XML生成"与"官方受理"。
> 5. **红队审查不是可选项。** 在递交前用四种角色攻击权利要求。
