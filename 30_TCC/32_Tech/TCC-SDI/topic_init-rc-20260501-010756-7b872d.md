# SMART Research Goal: Route≡Transform Isomorphism for TCC Primitive Set Completeness

---

## **Topic**
Route≡Transform: Formal Isomorphism between Communication Primitives (AllReduce, AllGather, Broadcast, AllToAll) and Computation Primitives (GEMM, Reduce, Scan, Map) on Reconfigurable Interconnect Fabrics for Topology-Centric Computing — with a focus on establishing **completeness and minimality** of the TCC-11 primitive set.

---

## **Novel Angle**

### What Has NOT Been Well-Studied

Existing work on collective communication (e.g., MPI collectives, TCCL, and their optimizations for GPU clusters) treats communication and computation as **fundamentally separate concerns** — the interconnect routes data, and the compute units transform it. Even recent co-design work (e.g., in-network computing with SmartNICs, P4-programmable switches, and near-memory processing) optimizes *individual* primitives in isolation, without establishing whether the **full set of primitives used is complete** (can express any distributed computation) or **minimal** (contains no redundant primitives expressible from others).

The specific gap is threefold:

1. **No formal algebraic isomorphism has been established** between the space of communication collectives and the space of computation primitives when the interconnect itself is reconfigurable (i.e., topology is not fixed). Prior work assumes static topologies (fat-tree, torus, hypercube) and derives communication schedules post-hoc. When the fabric is reconfigurable (optical circuit switching, RDMA-capable fabrics, silicon photonics crossbars), the boundary between "routing" and "computing" dissolves — a fact not yet exploited formally.

2. **The TCC-11 primitive set** (as proposed in emerging Topology-Centric Computing frameworks) has not been subjected to rigorous **completeness proofs** (can any distributed ML workload be expressed?) or **minimality analysis** (is there a smaller generating set?). This is analogous to the classical question of functional completeness in Boolean logic (e.g., NAND sufficiency), but for distributed dataflow — a question that has not been posed, let alone answered.

3. **Neuromorphic workloads** introduce sparse, event-driven communication patterns that do not map cleanly onto the dense collective primitives assumed by current TCC frameworks. Whether the TCC-11 set remains complete under sparse/neuromorphic workloads is entirely open.

### Why This Is Timely

- **Reconfigurable interconnects are entering production**: Optical circuit-switched fabrics (e.g., in hyperscaler AI clusters announced in 2023–2024) enable topology reconfiguration at microsecond timescales, making the compute/communicate boundary genuinely fluid for the first time.
- **LLM training at scale** has exposed that AllToAll (for Mixture-of-Experts routing) and AllReduce do not compose efficiently — practitioners are manually fusing them, suggesting a deeper algebraic structure is being discovered empirically rather than formally.
- **TCC as a paradigm** is gaining traction in systems research venues, but foundational questions about its primitive algebra remain unanswered, creating a first-mover opportunity for a formal treatment.

### How This Differs from Standard Approaches

Standard approaches: (a) profile collective performance on fixed topologies, (b) propose new algorithms for specific collectives, or (c) implement in-network compute for a single operation. This work instead asks a **structural/algebraic question**: given a reconfigurable substrate, what is the minimal generating set of primitives that spans the full space of distributed ML computations, and can we prove it?

## **Scope**

Focused on a single paper with the following boundaries:
- **In scope**: Formal algebraic framework for Route≡Transform isomorphism; completeness and minimality proofs (or counterexamples) for TCC-11; simulation-based validation on synthetic and ML workload traces; extension to sparse/neuromorphic patterns.
- **Out of scope**: Hardware implementation of a reconfigurable fabric; full-system benchmarking on physical clusters; runtime scheduling algorithms (left for future work).
- **Workload focus**: Transformer training (dense) + Spiking Neural Network inference (sparse) as two boundary cases.
