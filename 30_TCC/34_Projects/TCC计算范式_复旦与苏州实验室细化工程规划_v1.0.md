#### 1.2 TCC MLIR Dialect 定义

`mlir
// TCC Dialect 核心操作定义
tcc.cpc_map %input, %weight { op = "matmul", dtype = bf16 }
    → %result : tensor<M×K×bf16>

tcc.cpc_fold %input { op = "sum", axis = 0 }
    → %result : tensor<K×bf16>

tcc.cpc_scan %input { op = "cumsum" }
    → %result : tensor<N×bf16>

tcc.cpc_meld %input { op = "sum", axis = 1 }
    → %result : tensor<M×bf16>

tcc.npc_fuse %tensor across @devices { op = "sum", dtype = fp32 }
    → %result : tensor<N×fp32>

tcc.npc_swap %tensor across @devices { pattern = "uniform" }
    → %result : tensor<N×fp32>

tcc.npc_reduce %tensor to @root { op = "max" }
    → %result : tensor<N×bf16>

tcc.link_config %topology_id { pattern = "butterfly", size = 64 }
    → !tcc.topology

tcc.memory_alloc %size { buffer = "double", policy = "streaming" }
    → !tcc.buffer

// 复合操作（将被Pass 2分解为原语序列）
tcc.attention %q, %k, %v {
    heads = 32, dim = 128, causal = true
} → %output

tcc.fft_butterfly %input {
    points = 1024, direction = "forward"
} → %output
`

#### 1.3 拓扑调度Pass核心算法（Pass 4的核心创新）

`
算法: Minimum Topology Switch Scheduling

输入: NPC原语序列 [op1, op2, ..., opN], 每个op有其最优拓扑要求
输出: LINK.config调用序列 [cfg1, cfg2, ..., cfgM], M最小化

步骤:
1. 为每个NPC原语标注最优拓扑类型
   FUSE   → Butterfly / Tree / Ring
   SWAP   → Crossbar / All-to-All Mesh
   REDUCE → Tree
   BCAST  → Tree (反向)
   GATHER → Reverse Tree

2. 构建拓扑兼容图 G=(V,E)
   V = 每个NPC原语
   E(u,v) = 原语u和v可以使用相同拓扑结构执行

3. 图着色问题近似求解
   - 每种拓扑类型=一种颜色
   - 最小化颜色切换次数
   - 贪心+2-opt局部优化

4. 输出LINK.config序列
   - 连续同色原语共享同一拓扑配置
   - 只在颜色切换处插入LINK.config

示例:
  原语序列: [FUSE, FUSE, SWAP, SWAP, SWAP, FUSE, BCAST]
  最优调度: [Butterfly] [Crossbar]      [Butterfly]
            LINK(0)    LINK(1)          LINK(0)
  仅2次拓扑切换
`

**复杂度分析**：
- 图着色是NP-hard，但对NPU原语序列（通常<100步），贪心+2-opt可得近似最优解
- 拓扑切换本身<1μs，但编译器优化目标是**减少切换对流水线的打断**

#### 1.4 原语融合Pass的优化模式

| 模式 | 输入 | 输出 | 硬件收益 |
|------|------|------|---------|
| **MAP-FOLD融合** | matmul + reduce_sum | 融合矩阵乘法-累加器 | 消除中间张量写回，节省50%带宽 |
| **FOLD-BCAST融合** | allreduce + broadcast | 硬件蝶形网络同时完成 | 归约结果直接在蝶形网络广播 |
| **Attention模式** | QKV投影 + softmax + 加权求和 | 优化的attention原语序列 | 减少3次存储往返 |
| **FFT模式** | 多层蝶形网络 | 单次NPC-FUSE配置=FFT | FFT-AllReduce同构硬件复用 |
| **LayerNorm模式** | reduce_mean + sub + square + reduce_mean | 融合LayerNorm原语 | 单pass完成，减少2次存储 |

#### 2.2 零拷贝数据路径

`
传统NCCL路径 (4次拷贝):
  GPU VRAM → [1] CPU visible → [2] NIC bounce buffer
  → [3] InfiniBand → [4] Remote GPU VRAM
  延迟: ~50μs (含PCIe转换)

TCCL零拷贝路径 (0次拷贝):
  TCC SRAM → SDIoN DMA (直接地址映射) → Remote TCC SRAM
  延迟: ~50ns (硬件原语直达)

关键设计:
  • SDK注册内存: tcclRegisterBuffer(ptr, size)
  • 内核绕过: TCC Core直连SDIoN，不经Host CPU
  • 完成通知: 硬件中断→用户态eventfd (无内核往返)
`
