## 摘要

### 背景：AI自我进化的两条差异化路径  大模型需具备“工具使用+任务执行+自我升级”能力，2025年出现两类进化体系： - AgentEvolver（2025.11）：模型自生成任务/评分/策略，实现自我增强 - AlphaEvolve（2025.06）：在可程序化评估任务中，自动搜索更优算法

## 正文

# 🧠 AgentEvolver vs AlphaEvolve：两条 AI 自我进化路线的对比

**一文看清：一个是“靠自己创造任务的 Agent 自进化体系”，一个是“在人类可评估任务中自动发现更优算法的进化系统”。**

---

# **01 背景：为什么“AI 的自我进化”这么重要？**

大模型强，但“会用工具、会执行任务、会升级自己”才是真正的智能。  
过去两年，围绕“能不能让 AI 自己进化？”出现了两条截然不同的路线：

* • **AgentEvolver（2025.11）** —— 让大模型自己出题、自己打分、自己总结策略，最终自己变强。
* • **AlphaEvolve（2025.06）** —— 在用户提供可程序化评估的数学/算法/工程问题中，用 LLM + 进化方法自动发现更优的解法。

它们都叫“进化”，但定位、输入与能力完全不同。

---

# **02 研究团队与目标：一个进化 Agent，一个进化算法**

## 🧩 AgentEvolver：构建“自我成长型 Agent”

目标很清晰：

> **让大模型像人类一样“自己教自己”，不用人工写任务与 reward，也不用人工标注数据。**

团队研究的是 LLM 如何在真实工具链（文件操作、API 调用、代码执行等）中不断提升执行能力。

---

## 🔬 AlphaEvolve：自动发现数学/算法新成果

DeepMind 团队的目标来自完全不同的世界：

> **给出一个可自动计算评分的任务，让系统自动搜索出更强的数学构造、算法或工程优化方案。**

任务包括：

* • 矩阵乘法算法发现
* • 图论/组合构造
* • 几何打包
* • 调度算法优化
* • kernel 性能优化  
  （均为可程序化 evaluate 的问题）

AlphaEvolve关注的是**算法科学发现**，不是 Agent 能力。

---

# **03 方法差异：两种完全不同的“进化哲学”**

---

# **3.1 AgentEvolver：自生成任务、自生成奖励、自总结策略**

AgentEvolver 由三个关键闭环模块构成：

## **① Self-Questioning：自己出题（任务生成）**

LLM 观察自己的弱点 → 自动生成任务  
例：

* • “把 JSON 文件按特定 schema 整合成表格”
* • “判断目录下哪些文件与搜索条件匹配”

与 AlphaEvolve 不同，这里 **任务不是人定义的，是模型自己发现“需要学的技能”**。

---

## **② Self-Attributing：自己打分（奖励自动生成）**

LLM 对每一步操作进行 dense 贡献归因：

* • 哪步是 helpful
* • 哪步是 harmful
* • 哪步完成目标

这构成了 **完全无需人工的 dense reward + final reward**。

---

## **③ Self-Navigating：自己总结策略并复用**

LLM 自动从经验池中总结：

* • 子策略
* • 工具使用模板
* • 多步执行的结构化 workflow  
  最终通过 PPO 形成可复用的“技能库”。

---

## ✔ 小结

> **AgentEvolver 是“自己生成教材 → 自己考试 → 自己批改 → 自己升级”的自我学习体系。**

---

# **3.2 AlphaEvolve：在“人类给出的可自动评分任务”上，用 LLM+进化搜索更优解法**

AlphaEvolve 采用的是典型的“程序生成 → 自动评估 → 进化优化”流程。

核心组件：

## **① 用户提供 evaluator（评估函数 h）**

这一点必须强调：

> **AlphaEvolve 的奖励来自用户可程序化的 `h(solution)`，而非 LLM 自产生的自然语言评分。**

这一点在原文中多次强调，包括任务定义、评价流程、cascade 机制等。

可评估任务包括：

* • matrix multiplication rank
* • 极值点集构造
* • 图论 combinatorics
* • 调度 / kernel 性能  
  所有结果均能自动跑回分数。

---

## **② LLM 生成代码 candidate（program diff）**

LLM 不是 judge，而是 **proposal engine**：

* • 生成新程序 diff
* • 修改参数/结构
* • 迭代优化

LLM 用于“生成思路”，不是“打分”。

---

## **③ Evolutionary Search：大量并行候选 + 多代进化**

系统对候选执行：

* • mutation
* • crossover
* • selection
* • MAP-Elites 风格存储
* • 并行 evaluate

形成一个高度并行的搜索生态（literally “an evolution farm”）。

---

## ✔ 小结

> **AlphaEvolve 是“人类设定 What，系统自动搜索 How”。**  
> **它不适合开放任务，也不适合工具链 Agent；但在数学与工程优化上极强。**

---

# **04 任务类型：根本不同的世界**

| 维度 | AgentEvolver | AlphaEvolve |
| --- | --- | --- |
| 任务来源 | 自生成 | 人类定义 |
| 是否需要 evaluator | ❌ 不需要 | ✅ 必须有 |
| 本质能力 | 工具使用、任务执行、技能提升 | 数学/算法结构发现 |
| 工作流 | 自我学习闭环 | 程序搜索 + 严格评估 |
| 输出 | 更强 Agent | 新算法、新构造、新优化方案 |

一句话：**一个是“学会做事的人”，一个是“找到更好解法的科学家”。**

---

# **05 实验：两者的亮点与胜场**

---

## **AgentEvolver 实验亮点**

* • 在真实文件系统、API、代码执行环境中显著提升工具使用表现
* • 小模型也能通过自学习实现能力增益
* • 任务完全自动构建，无需人工设计
* • 对实际产品化 Agent 最有意义（企业场景直接受益）

---

## **AlphaEvolve 实验亮点**

全部来自原论文的真实突破：

* • **首次找到 rank-48 的 4×4 复矩阵乘法算法**（打破 56 年纪录）
* • 在点集 packing、组合构造、kissing number 中获得 SOTA structure
* • 改进 Google 内部工程系统（如调度启发式、kernel tiling 等）
* • 展示可大规模并行 + 严格数学验证的进化流程

这些成果是“数学与工程界都认可的硬结果”。

---

# **06 局限性：互补，而非竞争**

---

## **AgentEvolver 局限**

> **其打分器与任务生成依赖强大的 LLM，在任务空间混乱或模型能力不足时可能偏移。**  
> 不适合需要严格数学验证的任务。

---

## **AlphaEvolve 局限**

> **需要用户提供自动评分函数，只能处理 machine-gradable 的数学/算法/工程任务。**  
> 无法处理自然语言、工具配置、多步任务规划等开放式问题。

---

# **07 对 Agent 架构设计的启发**

## ✔ 从 AgentEvolver 得到的启示

未来 Agent 的核心不是 Prompt，而是：

* • 能否生成自己的任务
* • 能否自动评估自己
* • 能否提炼可复用技能
* • 能否在人类不干预的情况下持续变强

这意味着 Agent 架构应支持：  
**任务生成模块＋技能缓存＋自监督奖励生成器＋策略更新模块。**

---

## ✔ 从 AlphaEvolve 得到的启示

在 Agent 系统中，凡是：

* • 可自动写 evaluator
* • 有明确性能指标
* • 可并行搜索

都可以由 AlphaEvolve 式系统进行自动优化，例如：

* • Prompt 优化（可自动评估的任务）
* • DSL 转化器
* • 调参器
* • 代码片段生成
* • 工程运行时优化模块

---

# **08 最终一句话对比总结**

> **AgentEvolver 解决“让 Agent 自己变强”的问题，AlphaEvolve 解决“让算法变得更强”的问题。**

这两条路线互不覆盖，却共同推动 AI 进入一个真正“能自我改进”的时代。
