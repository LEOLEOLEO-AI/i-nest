### **🏗️ 背景与场景：Google为何选择拓扑重构**

**TPU 8系列战略调整**  
- **产品发布**：2026年4月Google Cloud Next大会发布第八代TPU，首次实现**训推分离**，分为TPU 8t（训练）与TPU 8i（推理）两款物理芯片。  
- **核心逻辑**：单一种类芯片无法同时满足大模型预训练与大规模智能体推理的差异化需求。  

**产业背景与技术瓶颈**  
- **推理瓶颈转移**：从算力转向**存储层级与通信效率**，数据传输代价远超计算代价（FP4乘加运算耗能约10飞焦耳，外部存储器读取能耗为计算的1000倍）。  
- **Boardfly拓扑目标**：专为TPU 8i设计，核心是降低芯片间通信**网络直径（跳数）**，优化推理场景下的**All-to-All通信模式**。  
- **技术渊源**：基于2008年John Kim提出的Dragonfly拓扑，针对**MoE路由**、**Chain-of-Thought推理**等现代AI工作负载做定制化改进。  

### **❌ 问题与挑战：3D Torus为何在推理场景失效**

**3D Torus拓扑的局限性**  
- **训练场景适配性**：上一代Ironwood采用的3D Torus拓扑在训练时表现优异，匹配梯度同步、All-Reduce等**规则局部通信**模式。  

**推理场景的通信模式变革**  
| 变革因素 | 具体影响 |
| :------- | :------- |
| **MoE路由机制** | 每个token需调用分布在不同芯片上的任意专家模块，打破邻居通信限制，逐跳跳转导致延迟累积 |
| **Agentic AI场景** | 多模型协同、多轮交互使通信从"规则局部"转变为"随机全局"，网络直径成为关键瓶颈 |
| **3D Torus跳数极限** | 1000芯片规模集群中最大跳数达16跳，All-to-All通信延迟显著增加 |

### **🔬 技术：Boardfly的三层分层架构**

**层次化架构设计**  
采用**Building Block → Group → Pod**三层结构，每层使用不同物理介质和连接策略：  

| 层级 | 组成 | 连接方式 | 物理介质 | 规模 | 功能 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **Level 1: Building Block (BB)** | 4个TPU 8i芯片 | 环形全互联 | 板上内部ICI链路 | 4芯片/BB | 最小构建单元，提供16条外部链路 |
| **Level 2: Group (G)** | 8个BB | BB间全连接 | 铜缆 | 32芯片/Group | 本地化高速通信域 |
| **Level 3: Pod** | 36个Group | Group间全连接 | OCS光电路交换机 | 1,152芯片/Pod | 跨机架规模扩展 |

**路由逻辑与跳数优化**  
- **同BB内通信**：最多**2跳**  
- **同Group内跨BB通信**：最多**3跳**（源BB 2跳 + 跨BB 1跳）  
- **跨Group通信（最坏情况）**：**7跳**（源BB 2跳 + 出Group 1跳 + OCS 1跳 + 入Group 1跳 + 目标BB 2跳）  
- **优化效果**：相比3D Torus的16跳，跳数减少**56%**。  

**OCS光电路交换机配置**  
- **核心组件**：5台320×320端口OCS，每台连接所有36个Group的对应链路，实现全交叉互联。  
- **技术特性**：通过MEMS微镜阵列直接切换光信号，无需电信号转换，传输延迟接近物理极限。  
- **光模块比例**：光模块与TPU卡比例约为**1.25:1**。  

**Boardfly与Dragonfly的核心差异**  

| 对比维度 | Boardfly | Dragonfly |
| :------- | :------- | :------- |
| 底层拓扑 | 4芯片环形BB | 交换机+终端节点 |
| Group内连接 | 8块板卡铜缆全连接 | 全连接或Flattened Butterfly |
| Group间连接 | OCS光交换 | 直连电缆链路 |
| 路由机制 | CAE硬件加速同步 | UGAL自适应路由 |
| 优化目标 | MoE推理All-to-All通信 | 通用HPC |

### **📊 数据：性能提升与系统规格**

**网络指标改进**  

| 指标 | 改进效果 |
| :--- | :--- |
| 网络直径 | 从16跳→7跳，减少**56%** |
| All-to-All通信延迟 | 改善高达**50%** |
| 芯片间互联带宽 | 翻倍至**19.2 Tb/s** |

**TPU 8i系统规格（与Boardfly深度耦合）**  
- 片上SRAM：**384 MB**（Ironwood的3倍）  
- HBM容量：**288 GB**  
- Pod规模：**1,152芯片**（36 Group × 32芯片）  
- 每美元推理性能：提升约**80%**  

**CAE协同加速**  
- 集合通信操作（All-Reduce等）：延迟降低**5倍**  

### **🎯 总结：系统级设计思维的体现**

**核心结论**  
- **高基数设计方向**：没有单一拓扑能适配所有流量模式，拓扑选择需基于对流量模式的深刻理解。  
- **分层策略的工程智慧**：Group内部用廉价铜缆实现高密度直连，Group间通过OCS实现单跳互通，在带宽密度、延迟和成本间寻求最优平衡。  
- **多重优化协同**：Boardfly（芯片间通信延迟）、384MB片上SRAM（减少跨芯片数据搬运）、CAE加速引擎（全局同步操作）形成**多重覆盖的延迟优化体系**。  

### **📝 补充细节**
- **MoE路由**：混合专家模型推理中，每个token可能调用不同芯片上的专家模块，导致通信模式从局部转向全局。  
- **OCS光电路交换机**：通过物理层光信号直接切换，避免电信号转换延迟，是实现跨Group低延迟通信的关键。  
- **CAE硬件加速**：片上集合通信加速引擎，专门优化推理场景下的全局同步操作，与Boardfly拓扑形成软硬件协同。
