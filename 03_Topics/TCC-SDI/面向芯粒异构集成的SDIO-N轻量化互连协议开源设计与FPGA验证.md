先说说为什么让你在做先进封装的同时承接这个接口协议课题。先进封装解决的是"芯粒物理上怎么连"——bump pitch、interposer布线、热管理。但光有物理连接不够，芯粒之间"说什么语言、怎么说话、说多快"——这就是接口协议的事。做封装的人如果不懂协议，就像修路的人不懂交通规则，路修得再好也会堵车。反过来说，你比别人多一层理解：你知道物理层的真实约束是什么，延迟预算是怎么分配的，这是纯做RTL的人不具备的直觉。

**这个课题对你而言是"打通上下层"的桥梁**——向下连接你的封装物理研究，向上支撑课题组整个TCC（拓扑中心计算）验证平台。做成了，你就是同时理解物理层和协议层的稀缺人才。

课题定位是**轻量化**——不要求你做一个完整的工业级UCIe，而是做一个"够用、能跑、能验"的协议子集，用开源工具和AI辅助工具链来加速。工作量控制在业余时间可完成的范围内。

### 一、课题背景

课题组正在推进的TCC（Topology-Centric Computing）验证平台需要9类原语芯粒（5类通信原语NPC + 4类算子原语CPC）通过统一互连协议进行动态组合。这套协议我们命名为**SDIO-N（Software-Defined I/O for Network-centric Chiplets）**。

SDIO-N的核心特征是**四层架构**：

- **L1 物理层**：负责芯粒间的电气信号传输（SerDes/并行总线）
- **L2 链路层**：负责可靠传输（流控、错误检测、重传）
- **L3 协议层**：负责数据包的寻址、路由、仲裁
- **L4 语义层**：负责拓扑重构命令——**这是SDIO-N区别于UCIe等现有标准的核心创新**

你在这个课题中的任务是：**基于开源代码，设计并验证SDIO-N的L2-L4三层轻量化协议，在FPGA上跑通基本功能，并完成一致性测试套件。** L1物理层在FPGA验证阶段用Xilinx GTH收发器原语模拟，不需要你自己设计PHY。

### 三、研究内容

#### 3.1 M1阶段：开源代码调研与协议规格定义（第1-2个月）

**任务3.1.1——学习与调研（第1个月前两周）**

你需要先建立两方面的知识基础：

**(a) Chisel语言入门**

你的先进封装背景意味着你可能更熟悉Verilog/VHDL而非Chisel。但课题组的TCC平台基于Chipyard生态（Chisel），所以需要你掌握Chisel基础。以下是推荐的学习路径，按照优先级排列：

1. **《Digital Design with Chisel》**（Martin Schoeberl著，免费PDF）——用3-4天通读，重点看第2-5章（基本语法、组合逻辑、时序逻辑、状态机）
2. **Chisel Bootcamp**（GitHub: freechipsproject/chisel-bootcamp）——Jupyter Notebook交互式教程，花2-3个晚上跑完Module 1-3
3. **不需要精通Scala**——Chisel只用到Scala很小的子集，把它理解为"可以写循环和函数的Verilog"就够了

如果你更倾向于用SystemVerilog，也可以。但Chisel的参数化能力对协议生成器非常有优势——一套代码可以生成不同位宽、不同虚拟通道数、不同FIFO深度的协议实例。**建议至少尝试Chisel，最终选择权在你。**

**(b) 开源参考项目深度阅读**

你需要精读以下三个开源项目的代码和文档，不是泛泛浏览，而是能画出它们的模块框图和数据流图：

**项目1：Constellation NoC生成器**

- 仓库：`github.com/ucb-bar/constellation`
- 文档：`constellation.readthedocs.io`
- 重点理解：credit-based flow control（信用流控）、virtual channel（虚拟通道）、wormhole routing（虫洞路由）、packet格式定义
- 你需要回答的问题：Constellation的一个router模块有多少个端口？每个端口的信号列表是什么？credit是怎么回传的？死锁是怎么避免的？
- **这个项目是你L2+L3层设计的主要参考蓝本**

**项目2：TileLink协议（Chipyard内置）**

- 规格书：SiFive TileLink Specification v1.8.1（网上公开PDF）
- Chipyard文档第9章：TileLink and Diplomacy Reference
- 重点理解：Channel A/D（必选通道）的请求-响应模型、消息类型（Get/Put/AccessAck等）、Diplomacy参数协商机制
- **这个项目展示了"如何在Chisel中优雅地定义分层协议"**

**项目3：UCIe协议规范（公开部分）**

- 来源：[uciexpress.org/specifications（注册后可下载）](http://uciexpress.org/specifications%EF%BC%88%E6%B3%A8%E5%86%8C%E5%90%8E%E5%8F%AF%E4%B8%8B%E8%BD%BD%EF%BC%89)
- 重点理解：UCIe的分层结构（Physical Layer → Die-to-Die Adapter → Protocol Layer）、Flit格式、Link State Machine（LSM）、Retimer机制
- **你不需要实现UCIe，但需要理解行业标准是怎么做分层的，作为SDIO-N设计的参照系**

**输出要求**：完成一份**《开源互连协议技术调研报告》**（10页以内），包含三个项目的架构对比表、SDIO-N可复用的模块清单、不可复用需要自研的模块清单。

**任务3.1.2——SDIO-N协议规格文档v0.5（第1个月后两周至第2个月）**

在调研基础上，与我讨论后，撰写SDIO-N轻量化版本的协议规格文档。文档结构要求如下：

```
SDIO-N Lite Protocol Specification v0.5
├── 第1章 概述
│   ├── 1.1 设计目标（轻量化、可重构、开源友好）
│   ├── 1.2 与UCIe/TileLink/Constellation的关系
│   └── 1.3 术语定义
├── 第2章 L2链路层
│   ├── 2.1 Flit格式定义（Header Flit / Data Flit / Control Flit）
│   ├── 2.2 Credit-based流控机制（参考Constellation裁剪）
│   ├── 2.3 错误检测（CRC-8，轻量化选择）
│   └── 2.4 链路初始化与训练序列（简化版）
├── 第3章 L3协议层
│   ├── 3.1 地址空间定义（芯粒编号 + 端口编号）
│   ├── 3.2 消息类型（DataReq / DataResp / TopoCmd / TopoAck）
│   ├── 3.3 路由机制（源路由 vs 表路由，选择其一）
│   └── 3.4 仲裁策略（轮询 / 优先级，可配置）
├── 第4章 L4语义层（★核心创新）
│   ├── 4.1 拓扑重构命令集定义
│   │   ├── TOPO_QUERY：查询当前拓扑状态
│   │   ├── TOPO_SWITCH：切换预定义拓扑配置
│   │   ├── TOPO_LINK_UP / TOPO_LINK_DOWN：启用/禁用指定链路
│   │   ├── TOPO_BW_ALLOC：动态带宽分配
│   │   └── TOPO_BARRIER：拓扑切换同步屏障
│   ├── 4.2 命令编码格式（建议定长16字节）
│   ├── 4.3 状态机定义（IDLE → RECONFIG → VERIFY → ACTIVE）
│   └── 4.4 与SDI-Ctrl芯粒的接口定义
├── 第5章 时序约束与性能目标
└── 附录A 信号列表（端口级接口定义）
```

**特别注意**：L4语义层是SDIO-N的灵魂。现有的UCIe、TileLink、AXI等协议都没有"拓扑重构"的语义层——它们假设拓扑是静态的。SDIO-N的创新在于：协议本身包含了"改变网络拓扑"的命令能力。这在学术上是新颖的，在工程上是TCC范式的必要条件。

设计L4时的指导原则：

- 命令集要**极简**——第一版只定义5-6条命令就够了，以后可以扩展
- 状态机要**确定性**——任何状态转移都不能有歧义
- 重构过程要**原子性**——要么完成切换，要么回滚到原拓扑，不能卡在中间态

**任务3.2.2——L2链路层RTL实现（第3-4个月）**

从Constellation的源代码中提取以下模块作为起点：


| Constellation源文件                | 对应SDIO-N模块      | 你需要做的改动            |
| ------------------------------- | --------------- | ------------------ |
| `channel/Channel.scala`         | SDIO-N Flit收发通道 | 修改Flit格式为SDIO-N定义  |
| `channel/CreditedChannel.scala` | Credit流控模块      | 裁剪虚拟通道数（从N个减到2-4个） |
| `router/InputUnit.scala`        | L2接收端           | 添加CRC-8校验逻辑        |
| `router/OutputUnit.scala`       | L2发送端           | 添加CRC-8生成逻辑        |


**关键设计决策**（我已替你做出，你可以在实现中验证其合理性）：

- **流控机制选credit-based**，不选handshake-based。理由：credit-based在流水线中延迟更确定，且Constellation已有成熟实现，你不需要从零开始。
- **Flit宽度选64 bit**。理由：这是先进封装中Die-to-Die接口的典型位宽，与UCIe对齐，同时在FPGA上用GTH跑也方便。
- **虚拟通道数选2个**（VC0给数据流量，VC1给控制/重构命令）。理由：轻量化，且足以避免协议死锁。
- **CRC选CRC-8**而非CRC-32。理由：轻量化，足够检测单bit翻转，面积开销极小。

**LLM辅助使用指南**：

- CRC-8生成器/校验器的Verilog/Chisel代码——**直接让LLM生成**，这是完全规范化的逻辑，LLM生成质量很高
- Credit计数器的RTL——**让LLM生成骨架，你修改细节**
- 不要让LLM生成完整的channel模块——你需要从Constellation代码中理解设计意图后自己改

**任务3.2.3——L3协议层RTL实现（第5-6个月）**

L3层核心是"地址解析 + 消息类型分发 + 路由表查找"：

- **地址格式**：`[芯粒ID: 4bit][端口ID: 4bit][偏移: 8bit]` = 16bit地址空间。支持最多16颗芯粒 × 16个端口——对于TCC验证平台绰绰有余。
- **消息类型编码**：

```
MessageType [3:0]:
  0000 = DataReq     （数据请求）
  0001 = DataResp    （数据响应）
  0010 = TopoCmd     （拓扑重构命令，L4语义）
  0011 = TopoAck     （拓扑重构确认）
  0100 = CreditReturn（信用回传）
  0101 = LinkTrain   （链路训练）
  0110-1111 = Reserved
```

- **路由选择**：第一版用**源路由（Source Routing）**。理由：实现最简单——发送方在Header Flit中写好完整路径，每个中间节点只需读取下一跳并转发。这比表路由少一个查找表模块，适合轻量化版本。后续版本可以升级为表路由。

**任务3.2.4——cocotb验证套件（与RTL并行进行）**

验证与设计必须同步进行，不是"写完RTL再写测试"，而是"先写测试再写RTL"（这叫测试驱动开发，在芯片验证领域同样适用）。

开源参考：`github.com/alexforencich/cocotbext-axi` 展示了如何用cocotb为AXI协议搭建完整的验证环境，包括Master/Slave BFM（Bus Functional Model）、随机激励、覆盖率收集。你的SDIO-N验证套件可以参照这个结构。

**LLM辅助使用指南**：

- cocotb测试用例——**大量使用LLM生成**。将你的Flit格式定义和消息类型定义给LLM，让它生成：单包传输测试、背靠背突发测试、credit耗尽-恢复测试、CRC错误注入测试、乱序到达测试等。每类测试让LLM生成10-20个变体。
- BFM（总线功能模型）——**让LLM生成SDIO-N Master BFM和Slave BFM**的Python类。BFM是发送/接收协议包的软件模型，本质上就是按照协议格式组装和解析数据。
- **你自己写的部分**：测试场景的优先级排列、边界条件的识别、覆盖率目标的定义。这些需要你理解协议才能做对。

#### 3.3 M3阶段：L4语义层实现与仿真（第7-9个月）

**这是整个课题最有创新性的部分。**

L4语义层没有任何开源项目可以直接参考——这是我们课题组的原创设计。但实现方法可以参考开源RISC-V的自定义扩展思路：将拓扑重构命令编码为定长协议包，用一个有限状态机（FSM）解析和执行。

**任务3.3.1——拓扑重构命令编解码器**

命令包格式（16字节 = 128 bit）：

```
SDIO-N L4 Topology Command Packet:
┌────────┬────────┬────────┬──────────────────────────┬────────┐
│ MsgType│ CmdCode│ SeqNum │ Payload (variable)       │ CRC-8  │
│ [3:0]  │ [7:0]  │ [7:0]  │ [95:0]                   │ [7:0]  │
├────────┼────────┼────────┼──────────────────────────┼────────┤
│ 0010   │见下表   │ 自增   │ 命令参数                  │ 校验   │
└────────┴────────┴────────┴──────────────────────────┴────────┘

CmdCode定义：
  0x01 = TOPO_QUERY    → Payload: [目标芯粒ID]
  0x02 = TOPO_SWITCH   → Payload: [目标拓扑ID: 8bit][生效模式: 即时/屏障]
  0x03 = TOPO_LINK_UP  → Payload: [源芯粒ID][源端口][目标芯粒ID][目标端口]
  0x04 = TOPO_LINK_DOWN→ Payload: 同上
  0x05 = TOPO_BW_ALLOC → Payload: [链路ID][带宽比例: 0-100%]
  0x06 = TOPO_BARRIER  → Payload: [参与芯粒位图: 16bit]
```

**任务3.3.2——拓扑重构状态机**

```
            ┌──────────┐
            │  IDLE    │◄────────────────┐
            └────┬─────┘                 │
                 │ 收到TOPO_SWITCH       │ 重构完成/回滚完成
                 ▼                       │
            ┌──────────┐                 │
            │ PREPARE  │ 锁定当前流量    │
            └────┬─────┘                 │
                 │ 流量排空完成           │
                 ▼                       │
            ┌──────────┐                 │
            │ RECONFIG │ 执行拓扑变更    │
            └────┬─────┘                 │
                 │ 变更完成              │ 超时/错误
                 ▼                       │
            ┌──────────┐           ┌─────┴────┐
            │ VERIFY   │──失败──→ │ ROLLBACK │
            └────┬─────┘           └──────────┘
                 │ 验证通过
                 ▼
            ┌──────────┐
            │ ACTIVE   │ 新拓扑生效
            └────┬─────┘
                 │ 稳定运行
                 ▼
            回到IDLE等待下一次重构
```

关键设计要求：

- PREPARE阶段必须等待所有in-flight（在途）数据包完成传输——这通过credit全部回收来判断
- RECONFIG阶段原子执行——修改路由表/链路使能寄存器
- VERIFY阶段发送一个探测包验证新拓扑可达性
- 任何阶段超时（可配置，默认1000 cycle）都触发ROLLBACK

**任务3.3.3——三场景拓扑切换测试**

在仿真中验证以下三种典型拓扑切换场景：


| 场景        | 初始拓扑             | 目标拓扑           | 验证要点         |
| --------- | ---------------- | -------------- | ------------ |
| HPC→AI推理  | 4芯粒全互连（AlltoAll） | 2+2分组流水线       | 切换过程无数据丢失    |
| AI推理→信号处理 | 2+2分组            | 蝴蝶树（Butterfly） | 响应时间 < 100ns |
| 信号处理→HPC  | 蝴蝶树              | 4芯粒全互连         | 带宽重新均匀分配     |


注意：你不需要真的有4颗芯粒的硬件——在仿真中用4个SDIO-N端点实例 + cocotb驱动即可。

#### 3.4 M4阶段：FPGA原型集成与一致性测试（第10-12个月）

**任务3.4.1——FPGA移植**

将Chisel生成的Verilog导入Vivado，在VCU128上实现。FPGA上的验证架构：

```
VCU128 FPGA内部：
┌─────────────────────────────────────────────────┐
│  ┌──────────┐    SDIO-N     ┌──────────┐        │
│  │ 虚拟芯粒A │◄────────────►│ 虚拟芯粒B │        │
│  │ (发送端)  │   L2+L3+L4   │ (接收端)  │        │
│  └────┬─────┘              └─────┬────┘        │
│       │                         │              │
│  ┌────┴─────┐              ┌────┴─────┐        │
│  │ Traffic  │              │ Traffic  │        │
│  │ Generator│              │ Checker  │        │
│  └────┬─────┘              └─────┬────┘        │
```

说明：

- "虚拟芯粒"就是一个简单的数据收发模块 + SDIO-N接口
- Traffic Generator由PC通过UART控制，发送指定模式的测试包
- Traffic Checker检查接收到的数据正确性，统计延迟和吞吐
- L1层在FPGA内部用简单的FIFO模拟（不用GTH），因为两颗"虚拟芯粒"在同一块FPGA上

如果时间允许，可以扩展到**两块FPGA板卡间通过GTH高速串行链路连接**，验证真实的L1物理层行为——但这不是必须的，作为加分项。

### 六、LLM辅助工具使用规范

为了确保效率和质量，明确以下使用规范：

**可以大量使用LLM的场景**：

- cocotb测试用例生成（LLM生成后你审查功能正确性）
- CRC-8等标准化逻辑模块的RTL生成
- 文档初稿撰写、格式排版
- 环境搭建脚本、Makefile、CI配置
- Bug调试时的错误日志分析

**需要你自己完成的核心工作**：

- SDIO-N协议规格的技术决策（Flit格式、状态机定义、命令集设计）
- L4拓扑重构状态机的设计与正确性论证
- 从Constellation源码中识别可复用模块并理解其设计意图
- 验证场景的设计和覆盖率目标的制定
- 最终的FPGA时序收敛（这需要工程经验，LLM帮不了太多）

**绝对禁止**：

- 直接复制LLM生成的RTL而不仿真验证就提交
- 让LLM生成协议规格的核心技术决策而不与我讨论
- 在论文/专利中使用LLM生成的内容而不注明辅助工具使用

### 七、与先进封装研究的协同

你做的先进封装研究和这个协议课题之间有一个天然的交汇点——**L1物理层约束如何影响L2-L4协议设计**。具体来说：

- 封装中的bump pitch决定了die-to-die信号线数量，从而决定了SDIO-N的物理位宽上限
- interposer的信号完整性（串扰、ISI）决定了L1的最高速率，从而决定了L2 credit管理的节拍
- 热管理约束可能导致部分链路降速，L4的TOPO_BW_ALLOC命令需要能应对这种情况

如果你在先进封装研究中获得了具体的物理层参数（比如"在硅interposer上，55μm pitch micro-bump可支撑4 Gbps/pin"），可以反过来校准SDIO-N协议的时序参数，使协议设计有物理依据。这种跨层融合的视角本身就是很好的论文素材。
