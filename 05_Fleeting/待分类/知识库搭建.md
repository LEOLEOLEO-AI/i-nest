## 一、系统架构总览

整个体系分三层，逻辑清晰、各司其职：

**底层：Obsidian Vault — "知识图谱层"。** 这是整个体系的单一事实源（Single Source of Truth）。所有核心知识资产——原语库规范、专利文档、项目指标、术语对照表、变更日志、文档依赖图——都以 Markdown 文件的形式存储在 Obsidian 中，通过双向链接（`wikilinks`）构成一张活的知识图谱。Obsidian 的 Graph View 天然可视化文档间的引用关系，Dataview 插件可以动态查询所有引用了某个指标值或专利名称的文件列表。这正是 Genspark Hub 缺失的"文档依赖图"能力。

**中层：GitHub/Gitee — "版本控制与协同层"。** 通过 Obsidian Git 插件，Vault 中的每一次修改都自动 commit 并 push 到远程仓库。每个 commit 都带有时间戳和变更摘要。当您修改了专利名称，Git diff 精确显示"从 A 改成了 B"；当您调整了片上 SRAM 指标，历史记录完整可追溯。这解决了 Genspark 缺失的"语义版本追踪"问题。选择 Gitee 可确保国内访问稳定性和数据主权。

**上层：Genspark Workspace — "AI 生产力层"。** 负责执行具体的创作任务：生成申报稿、排版 PPT、检索最新文献、对标国际数据。Genspark Hub 绑定 Obsidian Vault 的导出文件作为上下文，Custom Agent 按角色分工执行，Advanced Workflows 负责自动化流程。

三层之间的数据流是单向权威、双向同步的：Obsidian 是权威源 → 导出到 Genspark 作为上下文 → Genspark 产出回写到 Obsidian → Git 自动记录变更。

当您修改专利名称时，只需改 `02-专利文档/` 下的源文件，然后查看依赖图，就能精确知道哪些下游文档需要同步更新。这就是之前缺失的"DAG"。

## 三、AI 增强层的配置

### 3.1 本地 Graph RAG：Neural Composer + Smart Connections

安装两个 Obsidian 插件组合。Smart Connections 负责语义向量检索——您问"哪些文件涉及功耗指标"，它通过 Embedding 相似度找到所有相关笔记。Neural Composer 叠加 LightRAG 的知识图谱层——它不仅知道哪些笔记文本相似，更知道"专利名称"这个实体被"指南"和"申报稿"引用，"Route≡Transform"被"原语体系"和"FPGA验证"依赖。两层合在一起，就是前面说的"向量检索找近邻，图谱检索找因果"。

全部数据存在 Vault 本地的 `.neural_memory` 文件夹，随 Git 同步，不上传云端，完全满足知识产权安全要求。

### 3.2 Claude Code 集成

通过 `obsidian-claude-pkm` 框架将 Claude Code 接入 Vault。这带来四个关键能力：

**跨会话记忆。** Claude Code 的 `memory: project` 模式会在 Vault 中维护 `CLAUDE.md` 作为持久化记忆文件。每次会话结束时自动写入关键决策摘要，下次启动时自动加载。这就是 Genspark 里跨会话丢失的那部分记忆。

**Slash 命令自动化。** 自定义 `/update-patent` 命令：输入新专利名称 → 自动扫描依赖图中列出的所有引用文件 → 逐一替换 → 生成 diff 报告 → 自动 commit 到 Git。这就是之前说的"级联更新引擎"，但比 Genspark Advanced Workflows 更精准，因为它直接操作 Markdown 源文件。

**Goal-aligner Agent。** 可以定制一个专门的"指标一致性审计 Agent"，定期扫描所有文档中的量化指标，与 `指标基线表.md` 做交叉比对，标出任何不一致的地方（比如某个旧版文档里还写着 "SRAM ≥2 GB" 而非修订后的 "片上 ≥4 MB"）。

**自动 Git Commit。** 通过 PostToolUse Hook，Claude Code 每次修改文件后自动触发 `git add + commit + push`，变更即时同步到 Gitee。

## 四、与 Genspark 的协同工作流

整个工作流如下：

**日常知识维护在 Obsidian 中完成。** 更新原语定义、修改专利内容、调整指标数值、补充文献对标——所有这些操作都在本地 Vault 中进行，由 Git 自动版本化，由 Graph RAG 自动索引。

**生产性创作任务交给 Genspark。** 需要生成完整的申报稿 Word 文档？从 Obsidian 导出最新的核心文件（指标基线表、术语对照表、研究内容摘要），上传到 Genspark Hub。Hub 的 Custom Instructions 里写："所有指标以附件'指标基线表'为准，所有术语以'术语对照表'为准"。然后让 Genspark 的 Super Agent 或专用 Custom Agent 执行生成任务。

**成果回流到 Obsidian。** Genspark 生成的终版文档下载回本地，放入 Vault 对应目录，Git 记录变更。如果生成过程中发现需要修改某个底层定义（比如 Genspark 在检索最新文献时发现某个对标数据需要更新），先在 Obsidian 中更新源头，再重新导出到 Genspark。

**关键原则：Obsidian 是写入端，Genspark 是读取端。** 永远不要在 Genspark 里直接修改核心定义然后试图反向同步——那会导致版本混乱。Genspark 负责"用"知识生产内容，Obsidian 负责"管"知识的真实性和一致性。

## 五、具体搭建步骤（按周末两天可完成）

**第一天上午：Obsidian 环境搭建。** 安装 Obsidian，创建 Vault，按上面的目录结构建好文件夹。安装核心插件：Obsidian Git（连接 Gitee）、Dataview（动态查询）、Templater（模板引擎）、Smart Connections（语义检索）、Neural Composer（Graph RAG）。在 Gitee 上创建私有仓库，配置 SSH Key，测试 push/pull。

**第一天下午：知识迁移。** 把当前所有核心文档从 Word/PDF 转为 Markdown 放入 Vault。重点工作：建立双向链接（在指南中引用专利时用 `专利-正交原语集与拓扑融合变换` 语法）、编写 `指标基线表.md` 和 `文档依赖图.md`、设置 `术语对照表.md`。首次 commit 并 push 到 Gitee。

**第二天上午：AI 层配置。** 安装 Claude Code CLI，克隆 `obsidian-claude-pkm` 模板，运行 `/adopt` 将其适配到您的 Vault 结构。配置 Neural Composer 的 LightRAG 后端（如用 Mac 可本地跑 Qwen2.5-14B，否则用 Gemini 2.5 Flash 做首次 ingest）。首次全量索引，构建知识图谱。

**第二天下午：Genspark 对接。** 在 Genspark 中创建总控 Hub，从 Obsidian 导出核心文件上传。设置 Custom Instructions。创建 2-3 个 Custom Agent（指南编审、专利审查、文献对标）。测试一个完整的闭环流程：在 Obsidian 中修改一个指标 → Git commit → 导出到 Genspark → 让 Agent 重新生成对应段落 → 下载回 Obsidian → 再次 commit。

## 六、这套方案对您的研究体系意味着什么

回到您的核心研究命题：您坚信**物理网络的结构复杂度与环境复杂度之比超过阈值时将涌现智能**。有趣的是，这套知识管理体系本身就呈现出类似的结构特征——Obsidian 的双向链接网络就是您"知识资产"的物理拓扑，当这个网络的连接密度（文档间引用关系的丰富程度）超过某个阈值时，Graph RAG 的回答质量会出现非线性跃迁：它不再只是"找到相关文档"，而是能"发现您自己都没意识到的跨文档逻辑矛盾"。

这恰好是一个您可以亲手观察到的"涌现"案例。从工程角度看，这套 Obsidian + Git + Genspark 方案解决了跨会话记忆、版本追踪、依赖图管理和 AI 辅助级联更新四个核心问题。从研究视角看，它本身就是您的理论在信息系统层面的一个微型实验场。
