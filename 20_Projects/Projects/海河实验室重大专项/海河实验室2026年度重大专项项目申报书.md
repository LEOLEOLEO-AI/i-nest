## 第一部分：接口标准体系——NPC × CPC × SDI 三类芯粒互联规范

### 1.1 为何必须制定接口标准

本项目包含三类芯粒（NPC通信原语、CPC算子原语、SDI控制器），由不同团队分工设计，须在同一块板卡上协同工作。若无统一接口标准，将出现：
- 信号电平不兼容 → 需外加电平转换，增加延迟
- 时序约定不一致 → 数据错位，功能错误
- 协议语义不统一 → 软件栈无法统一抽象

**接口标准是本项目技术体系的"主干道"，也是后续产业化的核心专利壁垒。**

### 1.2 SDIO-N 接口标准（Software-Defined Interconnect for Native primitives）

#### 1.2.1 分层架构

```
┌─────────────────────────────────────────────────────────────────┐
│                   SDIO-N 接口标准分层                            │
├──────────────┬──────────────────────────────────────────────────┤
│  L4 语义层   │ 原语事务：SEND_PRIM(type, data, topology_cfg)     │
│              │ 类型：NPC_ALLREDUCE / NPC_ALLTOALL /              │
│              │       CPC_MAP / CPC_SCAN / ...                   │
├──────────────┼──────────────────────────────────────────────────┤
│  L3 协议层   │ 轻量事务协议（LTP）                               │
│              │ • 包头：8B（原语类型4b + 节点ID12b + 长度16b      │
│              │           + 序列号16b + CRC8 8b）                │
│              │ • 流控：基于信用（Credit-based），无需ACK          │
│              │ • 重传：仅端到端，中间节点不缓存                  │
├──────────────┼──────────────────────────────────────────────────┤
│  L2 数据链路 │ 64B Flit（最小传输单元）                          │
│              │ 奇偶校验 + 前向纠错（RS码，可选）                  │
├──────────────┼──────────────────────────────────────────────────┤
│  L1 物理层   │ 差分信号（LVDS），16 Gbps/lane                    │
│              │ 延迟：< 2ns（芯粒间），< 10ns（板间）             │
│              │ 功耗：< 1 pJ/bit                                 │
│              │ 连接器：兼容 UCIe 1.0 物理层电气规范              │
└──────────────┴──────────────────────────────────────────────────┘
```

#### 1.2.2 信号定义（芯粒管脚层）

| 信号组 | 信号名 | 方向 | 位宽 | 说明 |
|--------|--------|------|------|------|
| 数据通道 | DATA[255:0] | 双向 | 256b | 主数据总线，4×64b Flit |
| 控制通道 | PRIM_TYPE[3:0] | 输入 | 4b | 原语类型编码 |
| 控制通道 | NODE_ID[11:0] | 输入 | 12b | 源/目标节点ID，支持4096节点 |
| 拓扑配置 | TOPO_CFG[63:0] | 输入 | 64b | SDI拓扑配置寄存器 |
| 时钟 | CLK_IN / CLK_OUT | — | 1b | 源同步，随路时钟 |
| 流控 | CREDIT[7:0] | 双向 | 8b | 可用缓冲信用值 |
| 状态 | READY / VALID | 双向 | 1b | 握手信号 |
| 重构 | RECONFIG_REQ/ACK | 双向 | 1b | 拓扑重构请求与确认 |

| 编码 | 原语 | 芯粒 | 说明 |
|------|------|------|------|
| 4'b0001 | ALLREDUCE | NPC-AR | 蝴蝶树归约 |
| 4'b0010 | ALLTOALL | NPC-AA | 全连接路由 |
| 4'b0011 | REDUCESCATTER | NPC-RS | 流水分段聚合 |
| 4'b0100 | ALLGATHER | NPC-AG | 多播汇聚 |
| 4'b0101 | BROADCAST | NPC-BC | 稀疏树广播 |
| 4'b1001 | MAP | CPC-M | 矩阵乘脉动阵列 |
| 4'b1010 | REDUCE | CPC-R | 树形归约器 |
| 4'b1011 | SCAN | CPC-S | 并行前缀扫描 |
| 4'b1100 | TRANSFORM | CPC-T | 地址重排+查表 |
| 4'b1111 | SDI_RECONFIG | SDI | 拓扑重构命令 |

### 1.3 SDI控制器（SDI-Ctrl）的角色与接口

SDI控制器是三类芯粒的"指挥中枢"，独立设计为第10颗芯粒（或集成于主控FPGA）。

```
SDI-Ctrl 职责：
1. 接收上层软件（PyiNEST / NCCL插件）的原语调用
2. 解析当前任务通信模式（AllReduce / AlltoAll / ...）
3. 查询拓扑配置表（HPC模式 / AI模式 / 信号处理模式）
4. 向所有NPC/CPC芯粒发送 TOPO_CFG 配置字
5. 监控 READY/VALID 握手，管理流控信用
6. 统计性能计数器（延迟、带宽利用率）
```

### 2.2 数据流路径（三种场景）

#### 场景A：超算HPC（CFD/FFT/分子动力学）

```
外部数据 → PCIe → HBM2E（暂存）
                        ↓
                   CPC-M（局部矩乘，PDE迭代计算）
                        ↓
                   NPC-AR（AllReduce，边界Halo同步）
                        ↓
                   CPC-S（前缀扫描，累积统计量）
                        ↓
              SDI-Ctrl 判断：收敛？否→循环 / 是→输出
                        ↓
                   PCIe → 主机（结果回传）

SDI拓扑配置（HPC模式）：
- NPC-AR: Butterfly树，深度6，64节点
- CPC-M + CPC-S：流水线串联，无气泡
- 拓扑切换时间：NPC-AR ↔ NPC-RS < 100ns
```

#### 场景B：智算AI推理（L4推理 / 多路视频分析）

```
视频流输入 → PCIe → HBM2E（帧缓冲）
                        ↓
                   CPC-M（卷积/GEMM主力，ResNet/YOLO骨干）
                        ↓
              ┌─────────┴──────────┐
              ↓                    ↓
         CPC-R（局部归约）    CPC-T（BatchNorm/Softmax/GELU）
              ↓                    ↓
              └─────────┬──────────┘
                        ↓
                   NPC-AA（AlltoAll，MoE专家路由）
                        ↓
                   NPC-AR（AllReduce，多卡梯度同步）
                        ↓
                   NPC-AG（AllGather，结果广播）
                        ↓
                   PCIe → 主机（推理结果）

SDI拓扑配置（AI推理模式）：
- CPC-M：独占SDIO-N总线60%带宽（计算密集）
- NPC-AA：全连通路由，32路专家并发
- NPC-AR：环形AllReduce，4卡聚合
- 整体流水：CPC层 与 NPC层 并行流水，无等待
```

## 一、立项意义

### 1.1 战略背景：冯·诺依曼范式遭遇根本瓶颈

当前人工智能、超级计算、实时信号处理三大领域对算力的需求持续爆发，但支撑这些需求的计算架构在过去70年间没有发生范式级变革：无论是CPU集群还是GPU集群，均沿袭冯·诺依曼"节点中心"范式——**数据必须被搬运到计算节点才能被处理**。

这一范式在当前规模下暴露出三重不可调和的矛盾：

| 矛盾 | 量化表现 | 根因 |
|------|---------|------|
| **通信墙** | 万卡AI训练集群中，集合通信占总时间40-70% | 计算与通信分离，数据反复搬运 |
| **功耗墙** | 数据中心PUE持续恶化，训练单次成本超百万元 | 无效数据移动消耗大量能量 |
| **扩展墙** | 集群扩展效率随节点数增加持续下降（典型值0.6-0.7） | 固定拓扑无法适应动态通信模式 |

**根本原因**：现有体系将"复杂节点×简单网络"视为不可改变的前提，一切优化在此框架内展开，边际收益递减。

### 1.2 范式创新机遇：复杂网络×简单节点

本项目提出并验证**网络中心计算（Network-Centric Computing, NCC）**新范式：

> **将计算中心从节点迁移到网络——物理拓扑即计算。**

核心命题：任意分布式计算均可分解为**路由（Route）**与**变换（Transform）**两类操作。路由操作（集合通信：AllReduce / AlltoAll / ReduceScatter / AllGather / Broadcast）可由物理网络拓扑原生承担；变换操作（GEMM / 归约 / 扫描 / 重排）由极简计算节点完成。当9类通信-算子原语芯粒通过软件定义互连（SDI）动态组合时，**同样的硅面积可实现3-5倍的有效算力提升**。

这是自1945年冯·诺依曼体系结构以来，计算范式的第三次迁移：
- 第一范式（1945-2010）：CPU节点中心
- 第二范式（2010-今）：GPU/加速器集群——仍是节点中心
- **第三范式（本项目）**：网络中心——物理拓扑即计算

### 1.3 对天津市及国家的战略意义

**技术自主**：突破NVIDIA NVLink / InfiniBand互连封锁，掌握集合通信物理拓扑定义权——这是制约国产AI算力体系能否独立运行的最后一块短板。

**产业换道**：在芯粒（Chiplet）异构集成新赛道上，建立中国主导的**SDIO-N接口标准**，避免重蹈RISC-V时代"架构自主但生态依赖"的困境。

**科学前沿**：为网络时空协同复杂度（CST）理论提供工程验证平台，推动"复杂网络×简单节点"在超算、智算、信号处理三大场景的规律性认知，支撑后续国家重大专项布局。

### 任务一：SDIO-N接口标准制定与验证（Year 1，300万）

**研究内容：**

3. 在FPGA平台（Xilinx VCU128）上验证SDIO-N协议正确性

**关键指标：**
- SDIO-N物理层延迟：< 2ns（芯粒间）
- 协议开销：< 5%

| CPC-R | 树形归约，sum/max/min/avg | 1024元素/cycle | 4mm² |
| CPC-S | Brent-Kung并行前缀 | 1024元素/cycle（FFT复用）| 6mm² |
| CPC-T | 地址重排+查表 | Softmax/GELU近似，误差<0.1% | 5mm² |

**CPC-S特殊设计（★原创）**：利用FFT-AllReduce同构定理，CPC-S蝴蝶运算图与NPC-AR蝴蝶拓扑图论同构，**两颗芯粒共享物理数据路径**，节省约30%布线资源。

### 任务四：SDI控制器与三场景拓扑配置（Year 2，150万）

**研究内容：**
1. SDI-Ctrl芯片设计：原语调度 + 拓扑查找表 + 性能计数器
2. 三类场景拓扑配置文件（HPC模式 / AI推理模式 / 信号处理模式）
3. 场景切换协议：< 500ns热切换，无数据丢失

**拓扑配置文件示例：**

```yaml
# HPC_MODE.topo
primary_prim: NPC-AR          # 主通信原语：AllReduce
secondary_prim: NPC-RS        # 辅助：ReduceScatter
compute_chain: CPC-M → CPC-S  # 算子流水：GEMM → 累积统计
bus_alloc:
  NPC-AR: 50%                 # SDIO-N带宽分配
  CPC-M: 30%
  CPC-S: 20%
latency_mode: THROUGHPUT      # 吞吐优先（vs 延迟优先）

# AI_INFER_MODE.topo
primary_prim: CPC-M           # 主算子：矩阵乘
route_prim: NPC-AA            # 路由：AlltoAll（MoE专家分发）
norm_chain: CPC-R → CPC-T     # 归一化链：Reduce → Transform
latency_mode: DETERMINISTIC   # 延迟确定性优先

# SIGNAL_MODE.topo
primary_prim: CPC-S           # 主算子：FFT扫描
fft_allreduce_shared: true    # ★启用FFT-AllReduce同构路径
cfar_prim: NPC-RS             # CFAR滑窗：ReduceScatter
beamform_prim: CPC-M          # 波束成形：矩阵乘
latency_mode: REALTIME        # 实时确定性
```
