# 第一部分：战略背景与范式机遇——"顶天立地"的架构革命

## 1.1 计算架构70年未变之困局

自1945年冯·诺依曼确立"存储程序"概念以来，计算架构的核心范式始终如一：**计算在节点内完成，通信是节点间的附属操作**。这一"节点中心"范式在Moore定律的庇护下延续70年，但至2026年，三重趋势正同时将其推向物理极限：

**趋势一：大模型向边缘端渗透，通信瓶颈从数据中心蔓延至端侧。** Meta Llama 4系列覆盖8B至405B全谱系，8B模型可在边缘设备以INT4精度运行。当多路并发推理时，各节点中间特征融合（AllReduce）的开销占端到端延迟的40%-60%。传统SoC的固化片上网络（如2D Mesh）被设计为承载均匀流量的"通用公路"，无法按特定算法数据流优化——海量算力被无效搬运吞噬。

**趋势二：智能传感器融合对异构计算提出终极考验。** L4自动驾驶需同时处理8-12路摄像头、4-6路毫米波雷达、1-3路激光雷达。视觉推理依赖矩阵乘密集计算（GEMM），雷达信号处理依赖快速傅里叶变换（FFT）——两者的最优数据流拓扑截然相反，而传统芯片只能用一套固定互连同时应对两种需求。

**趋势三：国际竞争格局加速演变，互连技术成为博弈制高点。** NVIDIA通过NVLink 5 + NVSwitch 4构建GB300 NVL72的72-GPU私有互连生态，以封闭协议锁定高端AI训练市场。UCIe 2.0虽统一了芯粒间物理层标准，但**协议层以上仍是空白**——如何让不同厂商的芯粒协同计算，如何让物理网络随算法需求动态重构，这是中国可以抢占的战略窗口。

# 第二部分：Route-Transform理论体系——范式级创新

## 2.1 从"节点中心"到"拓扑中心"：计算架构的范式跃迁

传统计算架构的核心假设是**计算发生在节点内部，互连网络仅负责数据搬运**。该假设在稀疏通信场景下成立，但在密集通信场景中，通信开销主导系统性能。

TCC范式的核心颠覆：**将互连网络从"数据搬运工"提升为"计算执行者"**。当物理拓扑本身原生承载通信/变换操作时，数据传输的过程即是计算执行的过程——通信与计算在物理层面合二为一。

```
传统范式：  数据 → 节点A → [搬运] → 节点B → 计算 → 输出
                        ↑ 通信开销（无效能量）

TCC范式：  数据 → 节点A → [拓扑原生计算] → 输出
                        ↑ 通信即计算（零额外开销）
```

## 2.2 Route-Transform分解定理（Theorem 1）

**定理**：任意分布式张量计算任务 $\mathcal{C}$，可唯一分解为路由操作（Route, $\mathcal{R}$）与变换操作（Transform, $\mathcal{T}$）的交替复合序列：

$$\mathcal{C} = \mathcal{T}_k \circ \mathcal{R}_k \circ \cdots \circ \mathcal{T}_1 \circ \mathcal{R}_1$$

**物理意义**：该定理证明Route与Transform是分布式计算的两个正交基。当硬件分别加速这两类基元操作时，任意分布式计算均可映射到同一套硬件上。

## 2.3 Route ≡ Transform同构定理（Theorem 2）——理论核心

**定理**：特定类别的Route操作与Transform操作之间存在严格的图同构关系。

**"杀手级图表"——8对Route-Transform同构映射**：

| 变换操作 (Transform) | 路由操作 (Route) | 同构类型 | 共享拓扑 |
|---------------------|-----------------|---------|---------|
| FFT蝶形运算 | AllReduce（维度有序） | 图同构 $\mathcal{B}_k \cong \mathcal{A}_k$ | Butterfly |
| 矩阵转置 | AlltoAll | 置换同构 | Crossbar |
| 前缀和 | Scan（前缀归约） | 操作同构 | Binary Tree |
| Reduce（树形归约） | Reduce（树形通信） | 恒等映射（完全相同！） | Tree |
| 卷积滑窗 | Shift + Overlap通信 | 循环同构 | Ring/Torus |
| 注意力QK^T | AllGather(K) + 本地GEMM | 功能分解 | Star→Local |
| 排序网络 | Butterfly通信 | 图同构 | Butterfly |
| 稀疏矩阵×向量 | Scatter/Gather | 邻接同构 | 原始图拓扑 |

**这张表的震撼之处**：我们以为截然不同的"计算"与"通信"，在拓扑层面竟然是同一件事。

## 2.4 元拓扑与SDI化合键——拓扑可重构的代数基础

Route-Transform同构证明了"通信≈计算"的数学事实，但尚需回答一个更根本的问题：**物理拓扑的重构本身是否有一个可计算的代数系统？**

**3类元拓扑（不可约化基底）**：

| 元拓扑 | 图论表示 | 物理语义 | 对应模式 |
|--------|---------|---------|---------|
| 有向边 | $P_2$ | 点对点单向传输 | MOVE |
| 星型图 | $K_{1,n}$ | 一对多扇出/扇入 | CAST（广播）/ FUSE（汇聚） |
| 环型图 | $C_n$ | 循环流水线 | 流水线AllReduce |

**5类SDI化合键（拓扑合成操作）**：

| 化合键 | 图论操作 | 物理语义 |
|--------|---------|---------|
| 平行键 | 笛卡尔积 $G \square H$ | 正交维度独立扩展 |
| 交叉键 | Kronecker积 $G \otimes H$ | 分形自相似扩展 |
| 融合键 | 强乘积 $G \boxtimes H$ | 全连通合成 |
| 叠加键 | 图并集 $G \cup H$ | 异构拓扑共存 |
| 替换键 | 图替换 $G[H_v]$ | 层级化模块嵌入 |

**核心定理**：基于3种元拓扑和5种SDI化合键，可数学推演所有通信原语的最优物理拓扑。这意味着TCC的拓扑重构不是任意的，而是受一个闭合的代数系统严格约束。

## 2.5 拓扑自由能原理——拓扑的自组织优化

将统计物理引入网络拓扑设计，提出**拓扑自由能原理**：

$$\mathcal{S}[\mathcal{T}] = \int_{0}^{T} \left[ \underbrace{\frac{1}{2}\mu\left(\frac{d\mathcal{T}}{dt}\right)^2}_{\text{重构动能}} + \underbrace{V(\mathcal{T}, \mathcal{W})}_{\text{通信延迟势能}} \right] dt$$

**核心思想**：系统运行时的最优拓扑构型，是使"拓扑重构动能"与"通信延迟势能"所构成的网络拉格朗日作用量最小化的解。通过求解网络欧拉-拉格朗日方程，液态硬件在不同计算负载下自发寻找最优互连结构——为SDI控制器的运行时拓扑调度算法提供了严格的数学基础。

## 2.6 Kronecker分形扩展——破解互连扩展墙

分布式计算面临的核心工程困境是**互连扩展墙**——随着节点数N增加，通信开销呈$O(N)$甚至$O(N^2)$增长。

基于SDI化合键中的**Kronecker积（交叉键）**，TCC范式提供根本性解决方案——**分形拓扑扩展**：

```
Die级:    4×计算模块 ← Ring(C₄) 互连
                ↓ Kronecker积 ⊗
Chiplet级: 4×Die      ← Ring(C₄) 互连 → C₄ ⊗ C₄ 分形
                ↓ Kronecker积 ⊗
Board级:   4×Chiplet  ← Ring(C₄) 互连 → C₄ ⊗ C₄ ⊗ C₄
```

**核心性质**：每级互连规则完全相同（自相似性），通信复杂度增长呈理想的$O(\log N)$而非$O(N)$。从根本上破解大规模互连扩展墙。

## 2.7 TCC范式的学术演化定位

```
第一代：节点中心计算（Node-Centric, 1945-）
  "计算在节点，通信是附庸" → 通信墙瓶颈

第二代：拓扑中心计算（Topology-Centric, 2026-）
  "物理拓扑即计算，Route ≡ Transform" → 本项目 NCC-Edge
  消除通信瓶颈，一套硬件多场景自适应，计算与信号处理架构合二为一

第三代：涌现计算（Emergent Computing, 远期）
  "物理网络自发涌现计算与智能行为"
```

本项目为**第二代范式的首次系统性工程验证**。

## 3.1 TCC-11原语库——从"5+4"到三分类完备体系

TCC-11遵循三条设计原则：**代数完备性**、**正交最小性**、**硬件可映射性**（每个原语对应≤50K LUT的独立RTL IP核）。命名规则：统一前缀`tcc.`，后接四字母可发音动词助记符。

# 第四部分："3+1"液态硬件极简架构

## 4.1 架构设计哲学

TCC-11定义了三类原语。Phase 1的FPGA原型聚焦于**最小完备验证集**——恰好"3个数据原语 + 1个控制原语"：

- **tcc.FUSE**：通信原语。蝶形网络双模引擎，Route-Transform同构的物理证明载体
- **tcc.GEMM**：计算原语。脉动阵列矩阵乘，AI推理与DBF波束成形的共享算力源
- **tcc.MAPS**：计算原语。逐元素映射，FFT旋转因子、激活函数、归一化的统一执行单元
- **tcc.LINK**：流动原语。拓扑重构控制器，使物理网络从静态连线变为一级可编程资源

**设计原则**：原语级特化——每个IP核只加速一类操作，极致优化面积与功耗。LINK赋予硬件"柔性韧带"。

## 4.2 四核心IP深度设计

### IP-1: tcc.FUSE — 蝶形网络双模归约/变换引擎

**定位**：Route-Transform同构定理的物理证明载体。在同一蝶形网络硬件上，通过操作模式切换，同时实现AllReduce归约和FFT蝶形加减——**连线完全不变，仅改变节点内部操作类型**。

**双模硬件架构**：

```
             tcc.FUSE 蝶形网络引擎
        ┌─────────────────────────────────┐
        │  16路输入 × 256b                │
        │      ↓                          │
        │  Stage 0: 16→8                  │
        │  ┌──────────────────────┐       │
        │  │ 8个可配置运算单元      │       │
        │  │ 模式A: REDUCE_SUM    │       │  ← AllReduce归约
        │  │ 模式B: BUTTERFLY_ADDSUB│      │  ← FFT蝶形加减
        │  └──────────────────────┘       │
        │      ↓                          │
        │  Stage 1: 8→4 (同上)  Stage 2: 4→2  Stage 3: 2→1 │
        │      ↓                          │
        │  广播路径: 1→2→4→8→16（时分复用） │
        └─────────────────────────────────┘
```

**关键指标**：16节点FP16 AllReduce <2μs；1024点FFT蝶形吞吐<5μs；模式切换<100ns；RTL ~5,500行。

### IP-2: tcc.GEMM — 可配置脉动阵列矩阵乘引擎

**定位**：AI推理骨干和DBF波束成形的共享矩阵乘加速器。在线可切换阵列尺寸（32×32/64×16）和数值精度（INT8/FP16）。

**关键指标**：INT8 ≥1 TOPS（FPGA，等效ASIC 28nm约8 TOPS）；FP16 ≥250 GFLOPS；MAC利用率>80%；RTL ~6,000行。

### IP-3: tcc.MAPS — 逐元素映射引擎

**定位**：FFT旋转因子乘法、激活函数、归一化、阈值过滤的统一执行单元。TCC-11完备性验证中唯二全场景覆盖的原语。

**关键指标**：16路SIMD 256b/周期；支持≥15种FP16逐元素操作；延迟2-4周期；RTL ~4,200行。

### IP-4: tcc.LINK — 拓扑重构控制器

**定位**：液态硬件的"神经中枢"，TCC-Link标准的硬件实现载体。管理拓扑配置、切换与监控，使物理网络成为可编程资源。

**关键指标**：热切换≤1μs；≥16个预设拓扑；在线路由更新<100ns；RTL ~5,000行。

## 4.3 "3+1"完备性论证

| TCC验证目标 | 所需原语 | 验证方式 |
|---------------|---------|---------|
| Route-Transform同构 | FUSE + LINK | FUSE双模在同一LINK拓扑切换 |
| 通信硬件化 | FUSE | AllReduce在物理网络完成，CPU零参与 |
| AI推理全链路 | GEMM + MAPS + FUSE + LINK | 矩阵乘+激活+AllReduce+LINK管理 |
| DBF信号处理 | LINK + MAPS + FUSE + GEMM | FFT = LINK(butterfly) + MAPS(twiddle) + FUSE(butterfly)×10级 + GEMM(波束) |
| 液态切换 | LINK | ≤1μs AI↔DBF拓扑热切换 |

**结论**：4个原语构成Route-Transform同构验证的最小完备集。其余7个原语Phase 1 SDK软件回退，Phase 2 ASIC硬件化。

# 第五部分：TCC-Link接口标准——兼容与超越

## 5.1 标准定位

TCC-Link是本项目的核心标准输出。战略定位：**在UCIe尚未覆盖的协议层与语义层，建立由中国主导、原生支持拓扑可重构指令集的下一代芯粒互连标准。**

UCIe 2.0统一了物理层与链路层规范，但协议层仍是**点对点、静态互连**——定义"如何可靠传输数据"，未定义"如何让网络随算法变化"。TCC-Link填补这一空白：**"兼容并包，演进跨越"——物理层/链路层兼容国际标准，协议层/语义层实现范式级创新。**

```
UCIe 2.0（已完成）：                  TCC-Link v1.0（本项目）:
┌──────────────────────┐         ┌──────────────────────┐
│ 物理层 (L1)          │         │ 物理层 (L1)          │
│ 电气/管脚规范         │         │ ← 完全兼容 UCIe 2.0  │
├──────────────────────┤         ├──────────────────────┤
│ 链路层 (L2)          │         │ 链路层 (L2)          │
│ Flit格式/CRC/重传    │         │ ← 兼容UCIe + 扩展     │
├──────────────────────┤         ├──────────────────────┤
│ 协议层 (L3)          │         │ 协议层 (L3) ← 创新   │
│ 仅点对点传输          │  --->   │ 拓扑可重构指令集(TR-ISA)│
│ 无拓扑概念            │         │ 源路由 + 多跳         │
├──────────────────────┤         ├──────────────────────┤
│ (空白)               │         │ 语义层 (L4) ← 创新   │
│                      │         │ TCC原语事务语义        │
└──────────────────────┘         └──────────────────────┘
```

## 5.2 四层架构

**L1 — 物理层**：全面兼容UCIe 2.0 Advanced Package规范。16 GT/s (NRZ) / 32 GT/s (PAM4)，<2ns延迟，<1 pJ/bit。

**L2 — 链路层**：兼容UCIe Flit格式（64B），扩展1B拓扑控制字段。TopoCtrl全零时与UCIe标准格式完全一致，确保基本互通。

**L3 — 协议层（颠覆创新）**：**拓扑可重构指令集（TR-ISA）**——TCC-Link区别于所有现有互连标准的根本特征：

| 指令 | 操作码 | 功能 | 延迟 |
|------|--------|------|------|
| RECONFIG_TOPO | 0xF0 | 加载预设拓扑，并行更新全部受影响节点 | <500ns |
| RECONFIG_NODE | 0xF1 | 单节点路由表在线更新 | <100ns |
| RECONFIG_QUERY | 0xF2 | 查询节点路由表和链路状态 | <200ns |
| RECONFIG_ATOMIC | 0xF3 | 原子拓扑切换：冻结→批量更新→恢复 | ≤1μs |
| RECONFIG_TRAIN | 0xF4 | 触发链路重新训练（CDR/均衡器） | <10μs |
| RECONFIG_LOCK | 0xF5 | 锁定/解锁拓扑配置 | <50ns |

**TR-ISA指令以带内数据帧传输**——无需额外配置总线或JTAG链路。

**L4 — 语义层（颠覆创新）**：TCC-11原语事务接口。上层通过`tcc.fuse()`/`tcc.gemm()`等API调用，L4封装为标准事务帧。

## 5.3 标准输出路径

| 里程碑 | 时间 | 内容 |
|--------|------|------|
| v0.5 内部草案 | 2027 Q1 | L1-L4完整规范初稿，FPGA级验证 |
| v1.0 公开发布 | 2027 Q4 | 经FPGA验证修正的正式规范白皮书 |
| CCIA标准提案 | 2028 Q2 | 向中国通信工业协会提交立项 |
| UCIe对齐验证 | 2028 Q4 | v1.1，与UCIe 2.x规范对齐互操作 |
| 行业标准推进 | 2029+ | 基于ASIC验证的v2.0 |

# 第七部分：双场景融合验证——计算与信号处理"合二为一"

## 7.1 场景A：端侧多路并发AI视觉推理

**原语映射（AI模式）**：

```
# 每路K26运行YOLO-nano骨干
tcc.GEMM(im2col(frame), W_conv)     # 卷积
tcc.MAPS(SiLU(x))                    # 激活
tcc.MAPS(normalize(x, mu, sigma))    # BatchNorm

# 4路进入主FPGA融合
tcc.LINK.config("tree_4way")          # 蝶形归约树拓扑
tcc.FUSE(features, SUM)               # ★ 硬件AllReduce（CPU零参与）
```

| 指标 | 目标值 | 对比基线 |

| 4路AllReduce融合延迟 | <5μs | NCCL软件 ~100μs |

## 7.2 场景B：阵列雷达数字波束成形（DBF）——"合二为一"的震撼演示

**场景**：16通道雷达IQ数据，经FFT变换+波束成形，输出16个空间波束。**核心震撼点：执行AI推理的物理网络，与执行雷达DBF的物理网络，是同一组——实现计算架构与信号处理架构的"合二为一"。**

**原语映射（DBF模式）——FFT的TCC-11实现**：

```
# ★ FFT的核心洞见：FFT不是独立原语，而是 LINK+MAPS+FUSE 的组合！

for k in range(10):                     # 1024点 = 10级蝶形
    tcc.LINK.config(f"butterfly_stage_{k}")  # ← 每级配置蝶形拓扑
    tcc.MAPS(data[i] * twiddle[k][i])        # ★ 旋转因子逐元素乘
    tcc.FUSE(a, b, mode=BUTTERFLY_ADDSUB)    # ★ FUSE双模切换！蝶形加减

# 波束成形
tcc.LINK.config("crossbar_beam")             # 切换到波束级拓扑
tcc.GEMM(freq_data, beam_weights)            # 复数波束权重矩阵乘
```

**★ Route-Transform同构定理的现场证明——"合二为一"的物理实证**：

```
AI模式下的 FUSE:                    DBF模式下的 FUSE:
  FUSE(a, b, mode=REDUCE_SUM)        FUSE(a, b, mode=BUTTERFLY_ADDSUB)
     归约加法树                          蝶形复数加减
          ↓                                  ↓
    === 同一组物理蝶形连线，同一个 tcc.FUSE，同一个 tcc.LINK 拓扑 ===
          ↓                                  ↓
   AllReduce ≈ FFT  （在拓扑层面，它们是同一件事）
   
   ★ 计算架构 ≡ 信号处理架构 —— 一套硬件，两个世界，合二为一
```

**核心验证指标**：

| 指标 | 目标值 | 对比基线 |
|------|--------|---------|
| 1024点FFT延迟 | <5μs | 软件FFTW ~50μs |
| 全链路DBF延迟 | <20μs | 传统DSP方案 ~200μs |
| 物理路径复用率 | ≥30% | 独立实现需2套互连 |

## 7.3 场景C：液态切换压力测试

# 第八部分：软件栈与生态——即插即用的全栈SDK

## 8.1 PyiNEST-Lite 全栈SDK架构

TCC范式的成功取决于一个关键问题：**现有AI开发者能否零门槛使用？** PyiNEST-Lite的战略回答：**用户继续写PyTorch代码，底层自动映射到TCC原语硬件——即插即用，零代码改造。**

```
┌─────────────────────────────────────────────────────────────────┐
│              用户应用层（即插即用，零修改）                         │
│  import torch; model = torch.nn.DataParallel(model)              │
│  output = model(data)  # 代码完全不变                             │
├─────────────────────────────────────────────────────────────────┤
│            TCCL 兼容层（NCCL/MPI/Gloo/FFTW 多API映射）              │
│  ncclAllReduce(...) → tcc.fuse(buf, op=SUM, topo=BUTTERFLY)     │
│  MPI_Allgather(...)  → tcc.pull(buf)                             │
│  fftw_execute(...)   → for k: tcc.link+ tcc.maps + tcc.fuse    │
├─────────────────────────────────────────────────────────────────┤
│         PyiNEST-Lite Runtime（Python + C++）                       │
│  Graph Compiler: 原语调用序列优化为执行图                          │
│  Topology Planner: 自动生成最优拓扑切换序列（基于自由能原理）       │
│  Memory Manager: 片上SRAM/片外DRAM统一管理                        │
├─────────────────────────────────────────────────────────────────┤
│         PyiNEST-Lite Driver（C + 内核模块）                         │
│  PCIe Gen3 DMA引擎  |  tcc.LINK 寄存器映射                         │
│  中断/事件管理       |  Credit-based流控硬件接口                   │
├─────────────────────────────────────────────────────────────────┤
│            NCC-Edge FPGA 液态硬件                                  │
└─────────────────────────────────────────────────────────────────┘
```

**关键创新：TCCL多API兼容层**

TCCL实现NCCL API级兼容，拦截`ncclAllReduce()`等调用并翻译为TCC原语事务。同时兼容MPI collective、Gloo、FFTW——使**PyTorch、TensorFlow、JAX、MATLAB用户无需修改任何代码**即可直调液态硬件。即插即用覆盖≥90%的分布式训练/推理通信需求。

```python
# TCCL兼容层核心逻辑 —— 即插即用
class TCCLAllReduce:
    def __call__(self, sendbuff, recvbuff, count, datatype, op, comm, stream):
        topo_id = self.topo_selector.select(count, comm.nranks)
        if topo_id != self.current_topo:
            self.link.reconfig_atomic(topo_id)     # tcc.LINK触发拓扑切换
        return self.tcc.fuse(sendbuff, op=op)       # tcc.FUSE执行硬件AllReduce

# 用户代码完全不变，即插即用
import torch.distributed as dist
dist.all_reduce(tensor)  # ← 自动走 TCCL → tcc.FUSE 硬件加速
```

## 8.2 tcc Python API

```python
import tcc

# 通信原语
tcc.fuse(buf, op=tcc.SUM, topo=tcc.BUTTERFLY)   # AllReduce
tcc.pull(buf)                                     # AllGather
tcc.cast(buf, root=0)                             # Broadcast
tcc.swap(sendbuf, recvbuf)                        # AlltoAll

# 计算原语
tcc.gemm(A, B, C, alpha=1.0, dtype=tcc.INT8)     # 矩阵乘加
tcc.fold(buf, op=tcc.ARGMAX)                      # 向量归约
tcc.maps(buf, op=tcc.SILU)                        # 逐元素映射
tcc.scan(buf, op=tcc.SUM)                         # 前缀扫描

# 流动原语
tcc.link.config(tcc.BUTTERFLY, stage=3)           # 拓扑控制
tcc.move(src, dst, nbytes)                        # 点对点搬移
tcc.tick.stamp()                                  # 同步屏障
```
