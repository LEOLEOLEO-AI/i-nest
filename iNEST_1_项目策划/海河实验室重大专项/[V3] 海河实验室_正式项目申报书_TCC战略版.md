# 第一部分：战略背景与范式机遇

## 1.1 计算架构70年未变之困局

自1945年冯·诺依曼提出"存储程序"概念以来，计算架构的核心范式始终未变：**计算在节点内完成，通信是节点间的附属操作**。这一"节点中心"范式在过去70年间支撑了摩尔定律的持续兑现，但时至2026年，三大趋势正在将其推向极限：

**趋势一：大模型向边缘端渗透，通信瓶颈从数据中心蔓延至端侧。** Meta Llama 4 系列已覆盖从 8B 到 405B 的全谱系，8B 模型可在边缘设备上以 INT4 精度运行，但多路并发推理时，各节点的中间特征融合（AllReduce）开销占端到端延迟的 40%-60%。传统 SoC 的固化片上网络（如 2D Mesh）对此无能为力——它被设计为承载均匀流量的"通用公路"，而非针对特定算法数据流的"专用高铁"。

**趋势二：智能传感器融合对信号处理提出异构计算需求。** L4 自动驾驶需同时处理 8-12 路摄像头、4-6 路毫米波雷达、1-3 路激光雷达的多模态数据。视觉推理需要矩阵乘密集计算（GEMM），雷达信号处理需要快速傅里叶变换（FFT）——两者的最优数据流拓扑截然不同，而传统芯片只能用一套固定互连同时应对两种需求，**如同用高速公路同时跑 F1 赛车和越野卡车**。

**趋势三：国际竞争格局加速演变，互连技术成为博弈制高点。** NVIDIA 通过 NVLink 5 + NVSwitch 4 构建了 GB300 NVL72 的 72-GPU 全互连生态（2025），以私有互连协议锁定高端 AI 训练市场。UCIe 2.0（2024）虽然统一了芯粒间物理层标准，但**协议层以上仍是空白**——如何让来自不同厂商的芯粒在同一块晶圆上协同计算，如何让物理网络随算法需求动态重构，这是 UCIe 尚未触及、也是中国可以换道超车的战略机遇。

## 1.3 海河实验室的战略机遇：SDSoW先导项目

海河实验室在"软件定义晶上系统（SDSoW）"方向上已有系统性布局。TCC范式是SDSoW的理论内核，SDIO-N是SDSoW的标准基础，"液态硬件"是SDSoW的工程表达。本项目作为**SDSoW的先导与奠基性项目**，承担三大战略使命：

**本项目不追求9颗芯粒的全定制流片，而是以1945年ENIAC验证"存储程序"概念的战略逻辑，用FPGA验证TCC范式。范式成立后再推进流片，分步降低风险。**

# 第二部分：TCC理论体系——范式级创新

## 2.1 从"节点中心"到"拓扑中心"：计算架构的范式跃迁

传统计算架构的核心假设是：**计算发生在节点（ALU/PE/Core）内部，互连网络仅负责数据搬运**。这一假设在稀疏通信场景下成立，但在密集通信场景（分布式Transformer推理、多传感器融合）中，通信开销主导系统性能。

TCC范式的核心颠覆：**将互连网络从"数据搬运工"提升为"计算执行者"**。当物理拓扑本身可以原生承载通信/变换操作时，通信操作不再需要经过节点——它在网络传输的过程中就已经完成了。

```
传统范式：  数据 → 节点A → [搬运] → 节点B → 计算 → 输出
                        ↑ 通信开销（无效能量）
TCC范式：  数据 → 节点A → [拓扑原生计算] → 输出
                        ↑ 通信即计算（零额外开销）
```

这一跃迁将计算架构从**第一代"节点中心"**推进至**第二代"拓扑中心"**，与从真空管到晶体管的范式转换具有同等深远的意义。

## 2.2 "路由-变换"二元分解定理

**定理（Route-Transform Decomposition）**：任意分布式张量计算的通信与变换操作，均可唯一分解为路由（Route）与变换（Transform）两类基元操作的有限组合。

```
分布式计算 ≡ Route(拓扑) ∘ Transform(算子)

Route 基元（5类）：ALLREDUCE | ALLTOALL | REDUCESCATTER | ALLGATHER | BROADCAST
Transform 基元（4类）：MAP | REDUCE | SCAN | TRANSFORM

完备性：{Route基元} × {Transform基元} 可表达任意分布式张量计算的前向与反向传播
```

**工程推论**：如果硬件可以实现这9类基元的物理加速，则任意分布式计算均可通过基元的组合映射到同一套硬件上——无需为每个算法定制芯片。

但与现有"指令集架构"思路的根本区别在于：**Route基元不由节点执行，而由物理互连网络原生承载**。这是一个全新的架构维度——拓扑本身成为可编程的计算资源。

## 2.3 FFT-AllReduce拓扑同构定理

这是TCC理论体系中最具工程冲击力的发现。

**定理**：在蝶形网络（Butterfly Network）拓扑上，AllReduce的归约树与 Cooley-Tukey FFT的蝶形计算图具有严格的拓扑同构关系。两者使用相同的节点连接模式，仅需切换节点内部的操作模式（加法归约 vs. 复数蝶形运算），即可在同一物理连线上实现两种截然不同的功能。

```
AllReduce归约 (AI推理)              FFT蝶形计算 (雷达DBF)

Stage 2: sum = a0+a1+a2+a3         Stage 2: X[0..3] = FFT(x[0..3])
         ↙    ↑    ↑    ↘                   ↙    ↑    ↑    ↘
Stage 1: a0+a1 → sum1  a2+a3 → sum2       butterfly → X[0,2]  butterfly → X[1,3]
         ↙  ↗              ↙  ↗            ↙  ↗              ↙  ↗
Stage 0: a0  a1             a2  a3         x0  x1             x2  x3

         ====== 物理连线完全相同 ======
```

**量化收益**：同一套蝶形连线和交换节点，在AI模式下执行 AllReduce 归约（FP16/INT8加法树），在DBF模式下执行 FFT 蝶形变换（复数乘法+加法）。SDI控制器仅需切换节点操作模式和控制数据流向，无需重新布线。预期节省 ≥30% 的互连资源（连线 + 交换节点）。

**学术定位**：该定理揭示了计算与通信之间存在尚未被充分认识的深层统一性——某些计算图（如FFT）本质上就是某种通信模式（如AllReduce）的"对偶"形式。这一发现为"拓扑-算法联合设计"开辟了全新的研究方向。

## 2.4 TCC范式的学术演化定位

```
第一代：节点中心计算 (Node-Centric Computing, 1945-)
  "计算在节点，通信是附庸"
  代表：冯·诺依曼架构, x86, ARM, GPU, NPU
  瓶颈：通信墙 (Communication Wall)

第二代：拓扑中心计算 (Topology-Centric Computing, 2026-)
  "物理拓扑即计算，通信与计算在物理层统一"
  代表：本项目 NCC-Edge 液态硬件
  突破：消除通信瓶颈，一套硬件多场景自适应

第三代：涌现计算 (Emergent Computing, 远期)
  "物理网络自发涌现计算与智能行为"
  关键：拓扑相变临界态 + 自适应集体动力学
```

本项目为**第二代范式的首次系统性工程验证**，旨在推动 TCC/CST 进入体系结构、高性能计算、信号处理三大领域的主流学术视野，并为第三代涌现计算提供物理实验平台基础。

# 第三部分：技术路线——三阶段演进战略

## 3.1 从FPGA原型到SDSoW的完整演进链

TCC范式的工程落地遵循"验证→产品化→规模化"的三阶段战略：

```
Phase 1 (本项目, 2026-2029): FPGA原型验证 —— 证明范式可行
  ├── "3+1"架构FPGA实现
  ├── SDIO-N v1.0标准发布
  ├── 双场景（AI+DBF）液态切换验证
  └── TRL-5 (技术就绪度：系统级原型验证)

Phase 2 (2029-2031): ASIC产品化 —— 端侧双模芯片
  ├── 基于验证IP的28nm/12nm ASIC流片
  ├── NCC-Edge-ASIC 产品原型（RT-Infer-30 边缘推理卡）
  ├── 孵化公司独立运营
  └── TRL-7 (技术就绪度：工程样机)

Phase 3 (2031-2035): SDSoW晶圆级 —— 大规模液态计算
  ├── 晶圆级复杂拓扑互连（连接规模 >10^8）
  ├── Chiplet + 3D堆叠异质集成
  ├── SDIO-N标准成为国内行业标准
  └── TRL-9 (技术就绪度：量产部署)
```

**本项目聚焦 Phase 1，但所有的架构决策（IP核接口、SDIO-N标准、软件栈API）均预留 Phase 2/3 的迁移路径。**

## 3.2 Phase 1 技术路线的关键决策

| 决策   | 选择                      | 理由                                           |
| ---- | ----------------------- | -------------------------------------------- |
| 验证载体 | FPGA（非ASIC流片）           | 规避流片风险，迭代周期从天缩短至小时级；FPGA天然可重构，是"液态硬件"的最佳物理载体 |
| 芯粒数量 | "3+1"（非9颗）              | TCC范式的最小完备验证集；9颗是产品化目标                       |
| 工艺节点 | FPGA（等效16nm）+ 可选28nm流片  | 28nm流片成熟可靠、成本可控，作为ASIC可行性预研                  |
| 场景数量 | 2主+1扩（非3主）              | AI推理和DBF是拓扑差异最大的两个应用；视频分析可被AI场景覆盖            |
| 标准策略 | L1/L2兼容UCIe + L3/L4颠覆创新 | 降低物理层风险，集中创新于协议/语义层                          |
| 软件栈  | NCCL兼容 + PyTorch零修改     | 最大化现有AI生态的即插即用性                              |

# 第四部分："3+1"液态硬件极简架构

## 4.1 架构设计哲学

传统的"大而全"通用计算核（如 x86 core、CUDA core、NPU PE）遵循"一个核做所有事"的设计哲学，导致每个核面积巨大、功耗高、利用率低。TCC范式遵循截然相反的设计哲学：

- **原语级特化**（Primitive-level Specialization）：每个IP核只加速一类基元操作，极致优化面积与功耗
- **通信硬件化**（In-Network Computing）：AllReduce 等 Route 基元由物理网络直接完成，不经过任何计算节点
- **拓扑可编程**（Topology as Software）：同一组物理连线在不同时刻构成不同拓扑，由SDI控制器在微秒级切换

## 4.2 四核心IP深度设计

### IP-1: NPC-AR — 硬件蝶形树 AllReduce 引擎

**定位**：TCC范式"通信硬件化"的核心证明载体。

**功能**：在蝶形树拓扑上，以流水线方式完成多节点数据的 AllReduce 归约操作。支持 FP16、INT8 两种数据精度的加法/最大值/按位或三种归约操作。

**微架构**：
```
输入: 16路 × 256b数据流
   │
   ├── Stage 0: 16→8 归约（8个归约器并行）
   ├── Stage 1: 8→4 归约（4个归约器并行）
   ├── Stage 2: 4→2 归约（2个归约器并行）
   ├── Stage 3: 2→1 归约（最终融合结果）
   │
   └── 广播路径: 1→2→4→8→16（AllReduce的share阶段）
         ↑ 与归约路径共用物理连线（时分复用）
```

**关键设计创新**：归约路径与广播路径共享同一组蝶形连线，通过 SDI-Ctrl 控制数据流向实现分时复用，节省约 40% 的连线资源。

**关键指标**：
- 16节点 FP16 AllReduce 延迟：<2μs
- 带宽利用率（有效数据/总带宽）：>85%（对比：PCIe方案约40%，传统软件AllReduce约15%）
- RTL规模：~5,500行 SystemVerilog
- 功耗：<3W（FPGA实测）

### IP-2: CPC-M — 可配置脉动阵列矩阵乘引擎

**定位**：AI推理的核心算力来源，也是DBF场景中波束权重矩阵乘的执行单元。

**功能**：可配置为 32×32 或 64×16 的二维脉动阵列（Systolic Array），支持 INT8（128 TOPS等效）和 FP16（32 TFLOPS等效）两种精度的矩阵乘-累加操作。

**微架构**：
```
      权重 (沿列方向广播)
      ↓  ↓  ↓  ↓
激活 → [MAC][MAC][MAC][MAC] → 部分和
(沿行   [MAC][MAC][MAC][MAC]
 方向   [MAC][MAC][MAC][MAC]
 广播)  [MAC][MAC][MAC][MAC]
              ↓  ↓  ↓  ↓
           部分和累加输出
```

**可配置性**：通过 SDI-Ctrl 指令在线切换阵列尺寸（32×32→64×16）和数据精度（INT8↔FP16），适应不同算法需求。AI推理场景使用 INT8 高吞吐配置；DBF场景使用 FP16 复数矩阵乘配置。

**关键指标**：
- INT8 峰值算力：≥1 TOPS（FPGA上，等效ASIC 28nm约8 TOPS）
- MAC 利用率：>80%（含数据加载流水线开销）
- RTL规模：~6,000行 SystemVerilog
- 功耗：<8W（FPGA实测）

### IP-3: CPC-S — 可重构蝶形FFT/Scan引擎

**定位**：FFT-AllReduce同构定理的物理验证载体，也是DBF场景的核心加速器。

**功能**：在同一组蝶形连线网络上，通过操作模式切换实现两种功能：
- **FFT模式**：流水线 Cooley-Tukey 蝶形 FFT（支持16-4096点，Radix-2/4）
- **Scan模式**：并行前缀扫描（Prefix Scan），用于稀疏注意力等算子的加速

**微架构**：
```
CPC-S 与 NPC-AR 共享蝶形连线网络

FFT模式：
  蝶形连线 → 复数乘法器 → 复数加法器 → 蝶形连线 → ... → 频域输出
  （每级连线之间插入复数蝶形运算单元）

Scan模式：
  蝶形连线 → 加法器（前缀累加）→ 蝶形连线 → ... → 前缀和输出

切换机制：
  SDI-Ctrl发送 RECONFIG_ATOMIC → CPC-S切换内部操作模式（<100ns）
  → 物理连线无需改变（这就是同构定理的工程价值）
```

**关键指标**：
- 1024点 FFT 延迟：<5μs（FPGA实测，对比软件FFTW约50μs）
- 与 NPC-AR 物理路径复用率：≥30%
- FFT↔Scan 模式切换延迟：<100ns
- RTL规模：~4,200行 SystemVerilog
- 功耗：<5W（FPGA实测）

### IP-4: SDI-Ctrl — 微秒级拓扑重构控制器

**定位**：液态硬件的"神经中枢"，负责物理拓扑的动态管理和微秒级热切换。

**功能**：
1. 存储≥16个预设拓扑配置（路由表 + 连线模式表）
2. 接收并执行 TR-ISA 拓扑重构指令
3. 管理 Credit-based 流控和端到端重传
4. 监控链路状态（误码率、延迟、温度）

**微架构**：
```
┌────────────────────────────────────┐
│         SDI-Ctrl 控制器             │
│                                    │
│  ┌──────────┐  ┌──────────────┐   │
│  │ 指令解码  │  │ 拓扑配置SRAM  │   │
│  │ (TR-ISA) │  │ (16×2048b)   │   │
│  └────┬─────┘  └──────┬───────┘   │
│       │               │           │
│  ┌────┴───────────────┴──────┐    │
│  │      拓扑执行引擎         │    │
│  │  - 路由表并行加载         │    │
│  │  - 连线模式原子切换       │    │
│  │  - 链路训练与校准         │    │
│  └────────────┬──────────────┘    │
│               │                   │
│  ┌────────────┴──────────────┐    │
│  │   流控与链路监控           │    │
│  │  - Credit-based flow ctrl │    │
│  │  - 误码率实时监测         │    │
│  │  - 温度/功耗感知          │    │
│  └───────────────────────────┘    │
└────────────────────────────────────┘
```

**关键指标**：
- 拓扑热切换延迟：≤1μs（从接收 RECONFIG_ATOMIC 指令到所有连线模式切换完成）
- 预设拓扑容量：≥16个完整配置
- 单节点路由表在线更新：<100ns
- RTL规模：~5,000行 SystemVerilog
- 功耗：<1W（FPGA实测）

# 第五部分：SDIO-N接口标准体系——兼容与超越

## 5.1 标准定位：中国主导的芯粒互连标准

SDIO-N（Software-Defined Interconnect for Native primitives）是本项目的核心标准输出。其战略定位是：**在UCIe尚未覆盖的协议层与语义层，建立由中国主导、原生支持拓扑可重构的下一代芯粒互连标准。**

UCIe 2.0（2024）的重大贡献在于统一了芯粒间物理层与链路层的电气/管脚规范，使得不同厂商的芯粒可以在物理上互连。但UCIe的协议层仍是**点对点、静态互连**——它定义了"如何可靠传输数据"（How），但没有定义"如何让网络随算法变化"（What if）。SDIO-N填补的正是这个空白。

```
UCIe 2.0（已完成）：                  SDIO-N v1.0（本项目）:
┌──────────────────────┐         ┌──────────────────────┐
│ 物理层 (L1)          │         │ 物理层 (L1)          │
│ - 电气规范           │         │ ← 完全兼容 UCIe 2.0  │
│ - 管脚定义           │         ├──────────────────────┤
├──────────────────────┤         │ 链路层 (L2)          │
│ 链路层 (L2)          │         │ ← 兼容 UCIe Flit格式  │
│ - Flit格式           │         │ + 扩展拓扑控制字段   │
│ - CRC/重传           │         ├──────────────────────┤
├──────────────────────┤         │ 协议层 (L3) ← 创新  │
│ 协议层 (L3)          │         │ - 拓扑可重构指令集   │
│ - 仅点对点传输       │         │ - 源路由 + 多跳      │
│ - 无拓扑概念         │  --->   │ - 带内重构信令       │
├──────────────────────┤         ├──────────────────────┤
│ (空白)               │         │ 语义层 (L4) ← 创新  │
│                      │         │ - 原语事务语义       │
│                      │         │ - 拓扑-算法映射      │
└──────────────────────┘         └──────────────────────┘
     静态互连标准                        动态液态互连标准
```

## 5.2 SDIO-N四层架构详解

### L1 — 物理层（兼容UCIe 2.0）

| 参数 | 规格 | 说明 |
|------|------|------|
| 信号制式 | 差分NRZ/PAM4 | 与UCIe 2.0 Advanced Package规范一致 |
| 单lane速率 | 16 GT/s (NRZ) / 32 GT/s (PAM4) | 满足FPGA SerDes（GTY/GTM）能力 |
| 延迟 | <2ns（芯粒间）/ <10ns（板间） | 端到端，不含协议开销 |
| 功耗效率 | <1 pJ/bit | 对标UCIe标准功耗包络 |
| 连接器 | 兼容UCIe 2.0管脚图 | 确保物理层互操作性 |

**策略**：L1完全采用UCIe 2.0成熟规范，确保物理层零风险、芯粒间物理互操作性，为上层创新提供坚实基础。

### L2 — 链路层（兼容+扩展）

在UCIe Flit格式（64B标准Flit）基础上扩展：

```
UCIe 标准Flit:  [Header: 2B] [Payload: 62B] [CRC: 2B]  = 66B
                     ↓
SDIO-N扩展Flit: [Header: 2B] [TopoCtrl: 1B] [Payload: 61B] [CRC32: 2B] = 66B
                            ↑ 新增字段
                 TopoCtrl:
                   - bit[0]: 路由模式(0=静态/1=动态源路由)
                   - bit[1]: 重构帧标记(0=数据帧/1=重构指令帧)
                   - bit[2:3]: 优先级(2b, 4级QoS)
                   - bit[4:7]: 保留(面向TR-ISA v1.1扩展)
```

**兼容性保证**：当TopoCtrl字段为全零时，Flit与UCIe标准格式完全一致，确保与UCIe兼容设备的基本互通。

### L3 — 协议层（颠覆创新）：拓扑可重构指令集（TR-ISA）

这是SDIO-N区别于所有现有互连标准的根本特征——在协议层原生定义了拓扑重构指令集：

| 指令 | 操作码 | 功能描述 | 执行延迟 |
|------|--------|---------|---------|
| `RECONFIG_TOPO` | 0xF0 | 加载预设拓扑ID，并行更新所有受影响节点的路由表和连线模式 | <500ns |
| `RECONFIG_NODE` | 0xF1 | 在线更新单节点的路由表条目（用于增量拓扑调整） | <100ns |
| `RECONFIG_QUERY` | 0xF2 | 查询指定节点的当前路由表和链路状态 | <200ns |
| `RECONFIG_ATOMIC` | 0xF3 | 原子拓扑切换：冻结数据流→批量更新→恢复传输，保证无数据丢失 | <1μs |
| `RECONFIG_TRAIN` | 0xF4 | 触发指定链路的重新训练（SerDes CDR锁定、均衡器校准） | <10μs |
| `RECONFIG_LOCK` | 0xF5 | 锁定/解锁拓扑配置（防止误操作） | <50ns |

**设计创新**：TR-ISA指令本身作为带内数据帧传输——无需额外的配置总线或JTAG链路。这意味着拓扑重构可以像发送数据包一样简单，从根本上降低了SDI的工程复杂度。

### L4 — 语义层（颠覆创新）：原语事务与拓扑-算法映射

L4定义了面向上层软件的原语事务接口：

```c
// 原语事务的C语言抽象（PyiNEST-Lite Driver API）
typedef enum {
    PRIM_ALLREDUCE  = 0x01,
    PRIM_ALLTOALL   = 0x02,
    PRIM_REDUCESCAT = 0x03,
    PRIM_ALLGATHER  = 0x04,
    PRIM_BROADCAST  = 0x05,
    PRIM_MAP        = 0x09,
    PRIM_REDUCE     = 0x0A,
    PRIM_SCAN       = 0x0B,
    PRIM_TRANSFORM  = 0x0C,
} prim_type_t;

// 原语发送接口
int send_prim(prim_type_t type,
              void *data, size_t len,
              topo_cfg_t *topo_cfg);

// 拓扑重构接口
int reconfigure_topo(topo_id_t topo_id);
```

上层软件（PyTorch/TensorFlow）通过TCCL兼容层间接调用这些API，无需感知底层TR-ISA指令的细节。

# 第七部分：双场景融合验证

## 7.1 场景A：端侧多路并发AI视觉推理

**场景定义**：模拟L4级自动驾驶或智能安防场景——4路摄像头同时输入，每路运行YOLO-nano目标检测，中间特征需要跨路融合（AllReduce），最终输出融合检测结果。

**系统拓扑（AI模式）**：
```
4×K26(摄像头+特征提取) → SerDes → VU13P#1(CPC-M加速) → NPC-AR硬件AllReduce → VU13P#2(后处理) → PCIe → 上位机显示

SDI拓扑预设: "AI_FUSION_4WAY"
  连线模式: 蝶形归约树（4叶→2中间→1根）
  数据流: 4路特征 → Stage0归约 → Stage1归约 → 融合输出
  操作模式: NPC-AR=归约, CPC-M=INT8脉冲阵列, CPC-S=空闲
```

**核心验证指标**：

| 指标 | 目标值 | 对比基线 | 测量方法 |
|------|--------|---------|---------|
| 4路AllReduce融合延迟 | <5μs | 软件NCCL AllReduce ~100μs | SerDes入口→出口 时间戳差分 |
| 通信带宽利用率 | >85% | PCIe方案 ~40% | 有效数据/(总带宽×时间) |
| 端到端推理延迟(P99) | <2ms | CPU+软件融合 ~10ms | 视频帧输入→检测结果输出 |
| 有效能效比 | ≥8 TOPS/W(等效) | NVIDIA Jetson Orin ~3.2 TOPS/W | FPGA实测功耗/等效INT8算力 |

**创新亮点**：
- AllReduce融合在物理网络中完成，CPU完全不参与融合通信——这是"通信硬件化"的最直观演示
- 4路视频的中间特征在蝶形树中逐级归约，每级归约与下一级特征提取流水线并行，消除传统方案的串行等

## 7.2 场景B：阵列雷达数字波束成形（DBF）

**场景定义**：模拟16通道相控阵雷达的数字波束成形处理——16通道IQ数据同时输入，经FFT变换到频域，再与波束权重矩阵相乘，形成16个空间波束。

**系统拓扑（DBF模式）**：
```
4×K26(16通道IQ数据注入) → SerDes → VU13P#1(CPC-S蝶形FFT) → CPC-M(复数矩阵乘) → VU13P#2(波束输出) → PCIe → 上位机显示

SDI拓扑预设: "DBF_16CH_1024PT"
  连线模式: 蝶形FFT网络（与AI模式共享同一组蝶形连线）
  数据流: 16通道IQ → 4级蝶形FFT → 频域 → 波束权重矩阵乘 → 16波束
  操作模式: NPC-AR连线被CPC-S复用为蝶形FFT路径
```

**核心验证指标**：

| 指标 | 目标值 | 对比基线 | 测量方法 |
|------|--------|---------|---------|
| 1024点FFT延迟 | <5μs | 软件FFTW ~50μs | IQ输入→频域输出 时间戳差分 |
| 全链路DBF延迟 | <20μs | 传统DSP方案 ~200μs | IQ输入→波束输出 |
| 物理路径复用率 | ≥30% | 独立实现需2套互连 | 复用连线/总连线(FPGA布局验证) |
| 模式切换(AI→DBF) | ≤1μs | 传统方案需换硬件 | RECONFIG_ATOMIC触发→DBF模式首数据输出 |

**创新亮点**：
- **FFT-AllReduce同构定理的实时演示**：在同一组蝶形连线上，先运行AI推理展示AllReduce归约，再一键切换至DBF模式展示FFT蝶形变换——物理连线完全不变
- **路径复用可视化**：上位机以动画形式展示连线从"红色（AI归约路径）"切换为"蓝色（DBF蝶形路径）"，连线结构完全相同，仅颜色和操作标注变化——直观证明同构定理

## 7.3 场景C：液态切换压力测试

**目的**：证明液态切换不是"实验室一次性演示"，而是可以在连续生产中稳定运行的工程能力。

**测试方案**：
- 循环：AI模式30秒 → RECONFIG_ATOMIC → DBF模式30秒 → RECONFIG_ATOMIC → 重复
- 总循环：≥1000次（连续运行约16.7小时）
- 测量：每次切换延迟、切换成功率、数据完整性（输入/输出校验）

**验收标准**：
- 切换成功率：≥99.99%（1000次中允许失败≤1次）
- 单次切换延迟：P99 ≤1μs，P99.9 ≤2μs
- 数据完整性：1000次循环中无数据丢失或错误
- 链路稳定性：1000次循环后SerDes误码率无退化（<10^-12）

# 第八部分：软件栈与生态——打通AI最后一公里

## 8.1 PyiNEST-Lite 全栈SDK

TCC范式的成功不仅取决于硬件能否实现液态切换，更取决于**软件生态能否让现有AI用户零门槛使用**。如果每一个使用液态硬件的开发者都需要学习新的编程模型，TCC将难以走出实验室。

PyiNEST-Lite SDK的策略是：**让用户继续写 PyTorch/TensorFlow 代码，底层自动映射到 TCC 硬件。**

```
┌───────────────────────────────────────────────────┐
│              用户AI应用层（零修改）                  │
│  model = torch.nn.DataParallel(model)              │
│  loss = model(data)  # 代码完全不变                │
├───────────────────────────────────────────────────┤
│            TCCL 兼容层（NCCL API 映射）              │
│  ncclAllReduce(...) → pynest.send_prim(            │
│      type=PRIM_ALLREDUCE, data=tensor,             │
│      topo=topo_cfg["AI_FUSION_4WAY"])              │
├───────────────────────────────────────────────────┤
│         PyiNEST-Lite Runtime（Python + C++）        │
│  - 拓扑生命周期管理: load/save/reconfig/monitor    │
│  - 原语异步调度: async dispatch, completion poll   │
│  - 性能遥测: latency/bandwidth/power dashboard     │
├───────────────────────────────────────────────────┤
│         PyiNEST-Lite Driver（C + 内核模块）          │
│  - PCIe Gen3 DMA引擎（P2P, 零拷贝）                │
│  - SDI-Ctrl 寄存器映射（MMIO）                     │
│  - 中断管理: 完成中断、错误中断、热插拔中断           │
├───────────────────────────────────────────────────┤
│            NCC-Edge FPGA 液态硬件                    │
└───────────────────────────────────────────────────┘
```

**关键创新：TCCL（TCC Communication Library）兼容层**

TCCL是NCCL（NVIDIA Collective Communications Library）的API级兼容实现。它拦截所有NCCL调用（`ncclAllReduce`, `ncclBroadcast`, `ncclAllGather`等），翻译为对应的SEND_PRIM原语事务，并自动管理拓扑配置的加载与切换。

```python
# TCCL兼容层核心实现逻辑
class TCCLAllReduce:
    def __call__(self, sendbuff, recvbuff, count, datatype, op, comm, stream):
        # 1. 根据通信规模和数据分布，选择最优拓扑预设
        topo_id = self.topo_selector.select(count, comm.nranks)
        
        # 2. 触发SDI拓扑重构（如果需要切换拓扑）
        if topo_id != self.current_topo:
            self.sdi_ctrl.reconfig_atomic(topo_id)
            self.current_topo = topo_id
        
        # 3. 发送原语事务——数据直接在物理网络中完成AllReduce
        return self.sdi.send_prim(PRIM_ALLREDUCE, sendbuff, recvbuff,
                                   count, datatype, op)
```

**兼容性保障**：TCCL实现了NCCL API的一个核心子集（AllReduce/Broadcast/AllGather/ReduceScatter），覆盖≥90%的PyTorch DDP/FSDP通信需求。对于未覆盖的API，TCCL自动回退到CPU软件实现（透明保底）。

## 8.2 DSE工具：拓扑-算法联合设计空间探索

DSE（Design Space Explorer）是一个基于Python的分析工具，为后续Phase 2的ASIC流片提供量化设计依据：

```
输入:
  - 算法描述: 张量计算图（ONNX/TorchScript）
  - 资源约束: 面积上限(mm²)、功耗上限(W)、带宽上限(GB/s)
  - 工艺库: 28nm/12nm PDK参数

分析引擎:
  - 拓扑搜索: 在蝴蝶树/Fat-Tree/Clos/Dragonfly等拓扑族中搜索最优
  - 调度优化: 数据流调度、流水线深度、缓冲区大小
  - 同构匹配: 识别算法中可复用的拓扑结构（如GEMM+AllReduce同构）

输出:
  - 最优拓扑配置: topo_id + 路由表 + 连线模式
  - 性能预估: TOPS、延迟、功耗、面积
  - 对比报告: vs. 固定拓扑基线
```

# 第十二部分：预期效益与战略意义

## 12.1 科学价值——推动计算架构范式跃迁

**范式级贡献**：本项目将为"拓扑中心计算（TCC）"范式提供全球首个系统性的工程验证。如果成功，其科学意义可类比于1945年ENIAC对"存储程序"概念的验证——以原型系统证明一个新范式的工程可行性，为后续大规模实现奠定基础。

**理论突破**：
- Route-Transform 二元分解定理的严格证明与实验验证，将"通信"与"计算"这对传统上割裂的概念在数学层面统一
- FFT-AllReduce 拓扑同构定理的实验验证，开辟"拓扑-算法联合设计"的新研究方向
- 为第三代"涌现计算"范式（物理网络自发涌现智能行为）提供物理实验平台

**学术影响**：预期在体系结构四大顶会（ISCA/HPCA/MICRO/ASPLOS）和信号处理顶刊（IEEE TSP）发表论文≥5篇，推动 TCC/CST 范式进入体系结构、高性能计算、信号处理三大领域的主流学术议程。

## 12.2 技术价值——建立自主标准与IP体系

**SDIO-N标准**：国内首个原生支持拓扑可重构的芯粒互连标准。策略性选择L1/L2兼容UCIe（降低产业采纳门槛），L3/L4实现范式级创新（抢占下一代言片互连标准话语权）。标准白皮书公开发布并提交至CCIA。

**"3+1"验证IP核库**：经FPGA充分验证的4个原语级IP核，RTL质量达到直接迁移至28nm/12nm ASIC流片的成熟度。开源≥20K行SystemVerilog代码，为SDSoW晶圆级集成提供可复用的构建块。

**NCCL兼容软件栈**：TCCL兼容层实现PyTorch零修改对接液态硬件，证明TCC范式可以无缝嵌入现有AI生态——这是TCC从实验室走向产业化的关键一步。

## 12.3 产业价值——为SDSoW建立技术底座

本项目不输出直接产品，而是输出**经过验证的技术底座**，为后续产业化提供四大资产：

| 资产 | 形态 | 在后续产业化中的作用 |
|------|------|-------------------|
| SDIO-N标准v1.0 | 公开白皮书 + CCIA提案 | 吸引产业链合作伙伴，建立生态话语权 |
| "3+1"IP核库 | 开源RTL + FPGA验证报告 | 直接迁移至ASIC流片，缩短Phase 2周期 |
| PyiNEST-Lite SDK | 开源软件 + API文档 | 降低应用开发者接入门槛 |
| 双场景验证数据 | 延迟/带宽/功耗/切换的量化数据集 | 投资人/客户的决策依据 |

**产业化路线图**（本项目的后续延伸，不纳入本项目的交付范围）：

```
Phase 1 (本项目, 2026-2029):  FPGA原型 → 证明范式
```

**市场规模**：边缘AI推理芯片市场预计2030年达$50B+。若TCC范式产品化成功，目标市场包括：L4自动驾驶感知融合（>$10B）、智能安防多路视频分析（>$8B）、相控阵雷达/通信一体化（>$5B）。
