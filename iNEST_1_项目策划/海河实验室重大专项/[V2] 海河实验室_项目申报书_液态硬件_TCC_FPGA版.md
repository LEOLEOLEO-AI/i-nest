## 一、项目概述与范式创新

### 1.1 核心命题

过去70年，计算架构始终遵循冯·诺依曼"节点中心"范式——计算在节点内，通信在节点间。当端侧大模型推理与宽带雷达信号处理等场景将通信带宽推向极限，固化的片上网络拓扑成为不可逾越的瓶颈：**大量算力与能耗浪费在无效的数据搬运中。**

本项目提出并验证一个根本性反问：**如果物理拓扑本身就能承担通信操作，是否可以彻底消除通信瓶颈？**

这就是 **拓扑中心计算（Topology-Centric Computing, TCC）** 范式的核心洞察——通过**软件定义互连（SDI）**让硬件互连网络随算法需求动态重构，使物理拓扑原生承担 AllReduce、FFT 等通信/变换操作，实现"物理拓扑即计算"。

### 1.2 液态硬件：TCC的工程表达

SDI 赋予硬件的核心能力是**拓扑可重构性**。传统芯片的互连网络是一次性"烧死"的——AI芯片的Mesh拓扑无法处理FFT的蝴蝶图数据流，反之亦然。而 SDI 充当了硬件架构的"柔性韧带"，使同一套物理资源可以在微秒级内切换为截然不同的拓扑结构：

```
            传统刚性硬件                      液态硬件（本项目）
       ┌─────────┬─────────┐          ┌─────────────────────────┐
       │ AI加速卡  │ DSP板卡   │          │    NCC-Edge 液态硬件板卡   │
       │ (固定Mesh)│(固定FFT)  │          │  ┌───────────────────┐  │
       │  算力×1  │  算力×1   │          │  │  统一FPGA资源池     │  │
       └─────────┴─────────┘          │  │  NPC-AR + CPC-M     │  │
           需要两套硬件                │  │  + CPC-S + SDI-Ctrl  │  │
                                      │  └───────────────────┘  │
                                      │  SDI重构 --> AI模式       │
                                      │  SDI重构 --> DBF模式       │
                                      └─────────────────────────┘
                                           一套硬件，两种身份
```

**本项目不做9颗芯粒的全定制流片，而是用FPGA天然的可重构性作为"液态硬件"的最佳物理载体，聚焦于验证TCC范式的工程可行性。**

### 1.3 为什么V2放弃全定制流片路线

| 维度 | V1路线（9芯粒全流片） | V2路线（FPGA原型验证） |
|------|----------------------|----------------------|
| 技术风险 | 5批MPW，任一失败即全局崩塌 | FPGA天然可重构，迭代周期以天计 |
| 经费需求 | 流片+封装实际需3000万+ | 验证流片可选，不依赖核心指标 |
| 时间可行性 | 3年无法完成9芯粒设计→流片→回片→debug循环 | 3年充裕，每个阶段有迭代余量 |
| TCC创新验证 | 被工程风险淹没，无法聚焦范式验证 | **100%聚焦TCC范式本身** |
| 产业化路径 | 依赖芯粒良率与封装产能 | FPGA原型→ASIC流片→产品化，分步可控 |

**V2的哲学**：1945年冯·诺依曼用真空管ENIAC验证"存储程序"概念，而非直接造晶体管计算机。我们同样用FPGA验证TCC范式，而非在范式未经验证时冒险流片。

## 二、核心创新：TCC理论体系

### 2.1 "路由-变换"二元分解定理（Route-Transform Decomposition）

**核心主张**：任意分布式张量计算的通信操作，均可分解为"路由（Route）"与"变换（Transform）"两类基元操作的组合。

```
分布式计算 = Route(拓扑) o Transform(算子)

Route 基元：ALLREDUCE / ALLTOALL / REDUCESCATTER / ALLGATHER / BROADCAST
Transform 基元：MAP / REDUCE / SCAN / TRANSFORM
```

当物理拓扑可以被软件定义重构时，Route基元由**网络拓扑原生承载**（无需节点参与计算），Transform基元由**算子原语硬件加速**。两者在物理层面合二为一。

### 2.2 FFT-AllReduce拓扑同构定理（核心理论突破）

**定理**：在蝶形网络拓扑上，AllReduce的归约树与FFT的Cooley-Tukey蝶形计算图具有严格的拓扑同构关系。

```
 AllReduce归约树          FFT蝶形计算图          物理路径复用
                      
 Stage 0: O--O O--O      Stage 0: O--O O--O      同一组物理连线
          | | | |               |\/| |\/|     
 Stage 1: O--O O--O      Stage 1: O--O O--O      同一组物理连线
          |   |                 | \ |          
 Stage 2: O---O          Stage 2: O--O            同一组物理连线

  归约方向：叶→根           计算方向：输入→输出         连线完全复用
```

**工程意义**：在液态硬件上，执行AI推理的AllReduce通信网络和执行雷达DBF的FFT计算网络，可以使用**完全相同的物理连线**。SDI控制器仅需切换连线端点的操作模式（归约 vs. 蝶形运算），无需重新布线。

**预期资源节省**：>=30%的互连资源（连线和交换节点）因同构复用而节省。

### 2.3 TCC范式的学术定位

本项目在计算架构演化史中的位置：

```
第一代：节点中心（Von Neumann, 1945）
  "计算在节点，通信是附庸" → 通信墙瓶颈

第二代：网络中心/TCC（本项目, 2026）
  "物理拓扑即计算，通信与计算在物理层统一"
  → 消除通信瓶颈，一类新架构范式

第三代：涌现智能（远期愿景）
  "物理网络自发涌现计算行为"
```

本项目为第二代范式提供首次系统性的工程验证，推动"拓扑中心计算（TCC）"进入主流学术视野。
## 三、"3+1"液态硬件极简架构

### 3.1 架构设计原则

摒弃传统"大而全"的通用计算核，遵循**原语级特化（Primitive-level Specialization）**：

- **不做万能ALU**：每个IP核只做一类操作，极致优化面积与功耗
- **通信硬件化**：AllReduce等操作不经过CPU，由物理网络直接完成
- **拓扑可重构**：同一组物理连线在不同时刻承担不同拓扑角色

### 3.2 "3+1"核心构件

```
┌─────────────────────────────────────────────────────────┐
│                NCC-Edge 液态硬件架构                      │
│                                                         │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐              │
│  │ NPC-AR   │  │ CPC-M    │  │ CPC-S    │  ← "3"原语IP │
│  │ 通信原语  │  │ 算子原语  │  │ 算子原语  │              │
│  │ AllReduce│  │ 矩阵乘    │  │ FFT/Scan  │              │
│  │ 硬件归约  │  │ 脉动阵列  │  │ 蝶形变换  │              │
│  └────┬─────┘  └────┬─────┘  └────┬─────┘              │
│       │             │             │                     │
│       └─────────┬───┴─────────────┘                     │
│                 │                                       │
│       ┌─────────┴──────────┐                            │
│       │    SDI-Ctrl        │  ← "1" 柔性韧带            │
│       │  拓扑重构控制器      │                            │
│       │  • 路由表动态映射    │                            │
│       │  • 连线模式切换      │                            │
│       │  • <=1us热重构       │                            │
│       └────────────────────┘                            │
└─────────────────────────────────────────────────────────┘
```

| IP核 | 功能 | 关键指标 | RTL规模 |
|------|------|---------|---------|
| **NPC-AR** | 硬件蝶形树AllReduce归约<br>支持FP16/INT8累加 | 16节点融合延迟<2us<br>带宽利用率>85% | ~5K行 |
| **CPC-M** | INT8/FP16脉动阵列矩阵乘<br>可配置32x32或64x64阵列 | 峰值算力>=1 TOPS(FPGA)<br>MAC利用率>80% | ~6K行 |
| **CPC-S** | 并行前缀扫描 + 蝶形FFT<br>复用AllReduce物理路径 | 1024点FFT<5us<br>与NPC-AR路径复用>=30% | ~4K行 |
| **SDI-Ctrl** | 微秒级拓扑重构<br>路由表动态映射<br>Credit-based流控 | 模式切换<=1us<br>支持>=16拓扑预设 | ~5K行 |
| **合计** | | | **~20K行RTL** |

### 3.3 为什么"3+1"就够了

"3+1"是TCC范式的最小完备验证集。9颗芯粒是产品化目标，不是范式验证的必要条件。

| 验证目标 | 所需构件 | 说明 |
|---------|---------|------|
| TCC范式核心主张 | NPC-AR | 物理网络直接完成通信操作 |
| 同构定理验证 | NPC-AR + CPC-S | 同一条物理路径上的双模式演示 |
| AI推理场景 | CPC-M + NPC-AR | 矩阵乘加速 + 硬件AllReduce融合 |
| DBF信号处理 | CPC-S + CPC-M | FFT蝶形变换 + 复数矩阵乘 |
| 液态切换 | SDI-Ctrl | <=1us的AI<->DBF拓扑热切换 |

## 四、SDIO-N接口标准（V2·FPGA适配版）

### 4.1 标准定位

**策略：L1/L2充分兼容UCIe（降低物理层风险），L3/L4实现范式级创新（抢占标准话语权）。**

```
┌──────────────────────────────────────────────────────┐
│                  SDIO-N v1.0 标准分层                  │
├──────────────┬───────────────────────────────────────┤
│  L4 语义层   │ 原语事务：SEND_PRIM(type, data, cfg)   │
│ (颠覆创新)   │ 拓扑指令：RECONFIG(topo_id, mapping)   │
├──────────────┼───────────────────────────────────────┤
│  L3 协议层   │ 轻量事务协议（LTP）                     │
│ (颠覆创新)   │ - 原语类型编码（4b，兼容扩展至16b）     │
│              │ - 拓扑路由头（源路由，支持16跳）        │
│              │ - 拓扑重构信令（带内重构指令帧）        │
│              │ - Credit-based流控，端到端重传          │
├──────────────┼───────────────────────────────────────┤
│  L2 链路层   │ 兼容UCIe Flit格式（64B），扩展拓扑控制字│
│ (兼容扩展)   │ CRC32 + 可选RS前向纠错                  │
├──────────────┼───────────────────────────────────────┤
│  L1 物理层   │ 兼容UCIe 1.0电气/管脚规范               │
│ (充分兼容)   │ 16 GT/s per lane，标准Advanced Package  │
└──────────────┴───────────────────────────────────────┘
```

### 4.2 关键创新：拓扑可重构指令集（TR-ISA）

SDIO-N区别于UCIe的根本特征——在协议层原生支持拓扑重构指令：

| 指令 | 编码 | 功能 | 延迟目标 |
|------|------|------|---------|
| RECONFIG_TOPO | 0xF0 | 加载预设拓扑（路由表+连线模式） | <500ns |
| RECONFIG_NODE | 0xF1 | 单节点路由表在线更新 | <100ns |
| RECONFIG_QUERY | 0xF2 | 查询当前拓扑状态与链路质量 | <200ns |
| RECONFIG_ATOMIC | 0xF3 | 原子切换（AI<->DBF），保证无数据丢失 | <1us |

### 4.3 标准输出路径

- **Year 1**：SDIO-N v0.5 内部草案（FPGA级验证）
- **Year 2**：SDIO-N v1.0 公开发布，向CCIA提交标准提案
- **Year 3**：SDIO-N v1.1 与UCIe 2.0对齐，形成兼容+跨越的演进路线

## 五、FPGA原型平台方案

### 5.1 硬件平台

| 组件 | 选型 | 数量 | 用途 |
|------|------|------|------|
| 主FPGA | Xilinx VU13P(VCU128)或Altera Agilex 7 | 2块 | NPC-AR + CPC-M + CPC-S + SDI-Ctrl全系统 |
| 辅助FPGA | Kria K26 SOM | 4块 | 模拟多节点端侧设备，产生4路视频流/雷达数据 |
| 互连 | FPGA SerDes(GTY/GTM)+ 自定义PCB背板 | 1套 | 板间高速互连（16 Gbps x8 lane） |
| 测试仪器(租赁) | 高速示波器(>=20GHz)、误码仪、逻辑分析仪 | 按需 | 信号完整性验证 |

### 5.2 FPGA资源估算与可行性

以Xilinx VU13P（VCU128）为参考（~1.7M LUTs, ~6.8K DSPs, ~100Mb BRAM）：

| IP核 | CLB LUTs | DSPs | BRAM | 占比 |
|------|----------|------|------|------|
| NPC-AR x4 | ~80K | ~200 | ~200 | 15% |
| CPC-M 32x32 | ~120K | ~1024 | ~300 | 22% |
| CPC-S x2 | ~50K | ~400 | ~150 | 10% |
| SDI-Ctrl | ~40K | 0 | ~50 | 8% |
| 基础设施(SerDes/DDR/PCIe) | ~100K | 0 | ~300 | 20% |
| **合计** | **~390K** | **~1624** | **~1000** | **~75%** |

**75%利用率在合理范围内，留有25%余量应对集成开销**。若资源紧张可降级为单个CPC-S和2个NPC-AR，仍可完成全部验证指标。

### 5.3 验证方案

```
验证层级：
L0: 单IP核功能仿真（cocotb + Verilator）-> 覆盖率>95%
L1: 双FPGA互联通信验证 -> 延迟/带宽达标
L2: 单场景系统集成（AI推理/DBF各自独立验证）
L3: 双场景SDI热切换验证 -> <=1us切换 + 路径复用演示
L4: 上位机可视化演示系统 -> 实时展示拓扑重构过程
```

## 六、双场景验证方案

### 6.1 场景A：端侧多路并发AI视觉推理

```
  摄像头1 --> K26 #1 ---+
  摄像头2 --> K26 #2 ---+
  摄像头3 --> K26 #3 ---+--> NCC-Edge主FPGA --> 上位机显示
  摄像头4 --> K26 #4 ---+    (AllReduce融合)

  流程：
  1. 每路K26运行YOLO-nano骨干网络（CPC-M加速）
  2. 4路中间特征经SerDes送入主FPGA的NPC-AR
  3. NPC-AR在硬件蝶形树上完成AllReduce融合
  4. 融合结果送回上位机显示——无需CPU参与融合通信
```

| 验证指标 | 目标值 | 对比基线 |
|---------|--------|---------|
| 4路融合延迟 | <5us | CPU软件AllReduce ~100us |
| 端到端推理延迟(P99) | <2ms | CPU软件融合 ~10ms |
| 通信带宽利用率 | >85% | 传统PCIe方案 ~40% |
| 能效比 | >=8 TOPS/W（等效FPGA） | NVIDIA Jetson ~3 TOPS/W |

### 6.2 场景B：阵列雷达数字波束成形（DBF）

```
  16通道 --> K26x4 --> NCC-Edge主FPGA
  射频数据           (FFT蝶形计算网络)
                      |
  流程：              ├── CPC-S: 1024点FFT蝶形变换
  1. 16通道IQ数据输入  │   （复用NPC-AR的物理连线）
  2. SDI-Ctrl将NPC-AR  └── CPC-M: 复数波束权重矩阵乘
     连线切换为FFT模式
  3. CPC-S完成蝶形FFT  --> 16个波束输出
  4. CPC-M完成波束成形
```

| 验证指标 | 目标值 | 对比基线 |
|---------|--------|---------|
| 1024点FFT延迟 | <5us | 软件FFTW ~50us |
| 全链路DBF延迟 | <20us | DSP方案 ~200us |
| 物理路径复用率 | >=30% | 独立实现需2套互连 |
| 切换时间（AI-->DBF） | <=1us | 传统方案需换硬件设备 |

### 6.3 场景C（扩展）：液态切换压力测试

在A/B场景之上，增加**连续切换压力测试**：

- AI模式运行30秒 --> SDI切换 --> DBF模式运行30秒 --> 循环1000次
- 验证：切换成功率>=99.99%，无数据丢失，无链路失锁
- 上位机实时显示当前拓扑结构与数据流向（可视化演示核心亮点）

## 七、软件栈与生态

### 7.1 PyiNEST-Lite SDK 架构

```
        ┌──────────────────────────────────┐
        │       用户AI应用（PyTorch/TF）      │
        ├──────────────────────────────────┤
        │     TCCL兼容层（NCCL API映射）      │
        │     allreduce -> SEND_PRIM(NPC_AR) │
        ├──────────────────────────────────┤
        │     PyiNEST-Lite Runtime           │
        │     - 拓扑管理(load/save/reconfig) │
        │     - 原语调度(async dispatch)     │
        │     - 性能监控(延迟/带宽/功耗)      │
        ├──────────────────────────────────┤
        │     PyiNEST-Lite Driver            │
        │     - PCIe DMA引擎                │
        │     - SDI-Ctrl寄存器映射           │
        │     - 中断/事件处理                │
        ├──────────────────────────────────┤
        │     NCC-Edge FPGA 硬件             │
        └──────────────────────────────────┘
```

**关键创新：NCCL API兼容层**

通过拦截 `ncclAllReduce()` 等NCCL调用，翻译为 SEND_PRIM 原语事务，使现有PyTorch分布式训练/推理代码无需修改即可直调液态硬件。这极大降低了TCC范式的验证门槛。

```python
# 用户代码无需修改
import torch
model = torch.nn.DataParallel(my_model)
output = model(input_data)  # allreduce 自动走 NPC-AR 硬件
```

### 7.2 DSE工具（Design Space Explorer）

开发基于Python的TCC架构探索工具：

- 输入：算法描述（张量计算图）、资源约束（面积/功耗/带宽）
- 输出：最优拓扑配置方案、性能预估
- 用途：为后续ASIC流片提供量化设计依据

## 十、风险分析与缓解措施（新增）

### 10.1 技术风险

| # | 风险描述 | 概率 | 影响 | 缓解措施 |
|---|---------|------|------|---------|
| R1 | FFT-AllReduce同构定理存在未预见的边界条件 | 中 | 高 | Year 1优先完成严格的数学证明与大规模仿真验证（>=1000节点），若发现边界条件则限定定理适用范围，不影响核心范式主张 |
| R2 | <=1us热切换在FPGA上无法稳定实现 | 中 | 中 | 分级目标：1us(理想) -> 10us(良好) -> 100us(可接受)。即使100us也远超传统方案（需换硬件），TCC创新依然成立 |
| R3 | SerDes链路在PCB上信号完整性不达标 | 中 | 中 | 预留替代方案：降低速率至8 Gbps（仍满足指标），或改用FMC+标准子卡（成熟方案） |
| R4 | FPGA资源超出预期，无法容纳全部IP | 低 | 中 | "3+1"最小配置仅需~50%资源。集成时若超出，优先降级CPC-M阵列规模（32x32->16x16），核心验证指标仍可达成 |

## 十一、预期效益

### 11.1 科学价值

- **范式级贡献**：首次提供TCC（拓扑中心计算）范式的系统性工程验证，为"物理拓扑即计算"提供实验证据链
- **理论推进**：FFT-AllReduce同构定理的严格证明与实验验证，开辟"拓扑-算法统一设计"新研究方向
- **学术影响**：预期发表顶级论文5+篇（目标会议：ISCA/HPCA/SC/DAC），推动TCC/CST进入主流体系结构学术视野

### 11.2 技术价值

- **SDIO-N标准**：国内首个原生支持拓扑可重构的芯粒互连标准v1.0，在L3/L4层实现超越UCIe的范式创新
- **"3+1"IP核库**：经FPGA充分验证的原语级IP核，直接可迁移至后续ASIC流片
- **开源RTL**：>=20K行高质量开源RTL代码，为SDSoW晶圆级集成提供原型参考

### 11.3 产业价值

| 阶段 | 产出 | 预期影响 |
|------|------|---------|
| 项目期内 | NCC-Edge液态硬件原型 | 吸引后续流片/产业化投资 |
| 项目结束后1年 | 基于验证IP的28nm ASIC测试芯片 | 技术就绪度TRL-5 |
| 项目结束后2年 | NCC-Edge-ASIC产品原型 + 孵化公司 | 端侧AI推理/DBF双模市场 |
| 远期 | SDSoW晶圆级液态硬件 | 下一代计算架构基础设施 |

**本项目是SDSoW（软件定义晶上系统）在海河实验室落地的先导与奠基性项目。**

### 11.4 人才培养

- 博士后出站：2名（TCC理论 + FPGA架构方向）
- 博士研究生：4名（体系结构/集成电路各2名）
- 硕士研究生：8名（RTL设计/软件栈/验证方向）
- 为后续产业化公司储备核心技术团队
