## 〇、核心重构逻辑

原方案的问题在于：9颗芯粒 + 22nm流片 + 三场景验证 + 标准制定，这是一个**国家重点研发计划级**的工作量分解。5人团队用3年做完这些，即使有AI加持也不现实——光流片调试一颗芯粒就可能吃掉一个人半年时间，9颗×2轮流片，后端设计人力完全不够。

重构的核心逻辑是Peter Thiel说的那句话：“What can you do in 10 years is not interesting. What can you do in 6 months that would make the next 10 years different?”

**MVP的使命不是证明一切，而是证明一件事：网络中心计算范式在端侧场景中确实优于节点中心范式。** 只要在一个具体产品上证明了这一点，后续的标准制定、多场景扩展、芯片流片才有坚实的工程基础和说服力。

## 一、产品定义：NCC-Edge 端侧智能感知处理板卡

### 1.1 产品形态

**一块PCIe半高卡**（或M.2模块），板载1-2颗中等规模FPGA，实现NCC架构的核心子集，面向两个端侧市场：


| 特征   | 定义                                               |
| ---- | ------------------------------------------------ |
| 产品名称 | **NCC-Edge-1**（NCC架构端侧验证原型）                      |
| 物理形态 | PCIe ×4半高卡 或 自定义评估板                              |
| 核心器件 | AMD/Xilinx Kria K26 SOM（ZU5EV） ×2 + 板间高速互连       |
| 面向场景 | **场景A**：多路视频AI推理（端侧智算） **场景B**：数字波束成形DBF（端侧信号处理） |
| 目标用户 | 自动驾驶感知模块Tier-1供应商 / 相控阵雷达前端处理方                   |
| 定价区间 | BOM成本 < 8000元（原型），量产目标 < 3000元                   |


### 1.2 为什么选这两个场景

**场景A（多路视频AI推理）和场景B（数字波束成形）**恰好是NCC范式的甜蜜区，因为它们有一个共同特征：**数据天然分布在多个节点，且节点间需要大量集合通信**。

场景A中，多路摄像头各自推理后需要AllReduce融合结果（多视角目标检测融合）。传统方案是每路推理结果搬到主CPU做软件融合——数据搬运延迟高、CPU成为瓶颈。NCC方案将融合操作直接在互连网络中完成（NPC-AR硬化），推理节点只做矩阵乘。

场景B中，多通道ADC采样数据需要做FFT + 波束成形（本质是矩阵乘）+ CFAR检测。传统方案是集中式FPGA处理——所有通道数据汇聚到一颗大FPGA。NCC方案让每个通道有自己的小处理单元（CPC-S做FFT），通道间融合由网络拓扑完成（NPC-AR/NPC-RS），利用FFT-AllReduce同构定理共享物理路径。

**关键洞察**：这两个场景在计算图层面高度同构——都是"分布式局部计算 + 集合通信融合 + 全局决策"的三段式流水线。NCC架构的优势在于用一套硬件、通过SDI拓扑重构，同时覆盖两个场景。如果用传统方案，你需要为AI推理和信号处理各做一套硬件。

### 1.3 对标竞品与差异化


| 指标   | Hailo-8 (端侧AI) | Coral Edge TPU | Xilinx ZCU104 (信号处理) | **NCC-Edge-1** |
| ---- | -------------- | -------------- | -------------------- | -------------- |
| 算力   | 26 TOPS        | 4 TOPS         | ~2 TOPS等效            | 4-8 TOPS       |
| 功耗   | 2.5-3W         | 2W             | 15W                  | <10W           |
| 能效   | 8.7 TOPS/W     | 2 TOPS/W       | 0.13 TOPS/W          | **>1 TOPS/W**  |
| 多路推理 | 不支持原生融合        | 单路             | 不支持                  | **4路原生融合**     |
| DBF  | 不支持            | 不支持            | 支持但独立设计              | **同一硬件覆盖**     |
| 场景切换 | 不支持            | 不支持            | 不支持                  | **<1μs热切换**    |

## 三、MVP精简：从9颗芯粒砍到3+1

### 3.1 原语芯粒精简逻辑

原方案9颗芯粒中，真正支撑两个场景所需的核心原语只有3颗：


| 原语       | 功能            | 场景A（AI推理） | 场景B（DBF）        | 保留/砍掉                 |
| -------- | ------------- | --------- | --------------- | --------------------- |
| NPC-AR   | AllReduce     | 多路推理结果融合  | FFT-AllReduce同构 | **保留（核心）**            |
| CPC-M    | 矩阵乘           | 卷积/全连接推理  | 波束成形权值乘         | **保留（核心）**            |
| CPC-S    | 并行前缀/FFT      | Softmax近似 | FFT频谱变换         | **保留（核心）**            |
| SDI-Ctrl | 拓扑控制          | 场景切换      | 场景切换            | **保留（必要）**            |
| NPC-AA   | AlltoAll      | MoE专家分发   | —               | 砍掉（端侧无MoE需求）          |
| NPC-RS   | ReduceScatter | 分布式训练     | CFAR            | 砍掉（MVP不做训练，CFAR用软件实现） |
| NPC-AG   | AllGather     | 参数同步      | —               | 砍掉（端侧无需）              |
| NPC-BC   | Broadcast     | 参数广播      | —               | 砍掉（端侧无需）              |
| CPC-R    | 树形归约          | 归一化       | —               | 砍掉（CPC-M可兼任）          |
| CPC-T    | 地址重排          | 激活函数      | —               | 砍掉（查表用LUT实现）          |


**最终保留：NPC-AR + CPC-M + CPC-S + SDI-Ctrl = "3+1"架构**

这3颗原语恰好对应NCC理论中最核心的分解：

> 任意分布式计算 = Route（NPC-AR承担）+ Transform（CPC-M / CPC-S承担）

### 3.3 "3+1"架构的理论完备性论证

“3颗原语够不够表达NCC范式的核心思想？”——够的。论证如下：

NCC核心命题是"任意分布式计算 = Route + Transform"。在端侧场景中：

**Route操作**：端侧最核心的集合通信是AllReduce（多节点结果融合），NPC-AR覆盖。AllGather/Broadcast/ReduceScatter都可以用AllReduce的子操作组合实现（这在MPI理论中已有严格证明——Thakur et al., “Optimization of Collective Communication Operations in MPICH”, IJHPCA 2005, 被引2000+次）。所以NPC-AR在端侧是**通信充分的**。

**Transform操作**：端侧最核心的两类计算是矩阵乘法（CPC-M覆盖CNN/Transformer推理的核心算子）和频域变换（CPC-S覆盖FFT/前缀累积）。二者组合可覆盖端侧AI推理和信号处理中>90%的计算量。

**FFT-AllReduce同构**：CPC-S与NPC-AR共享蝴蝶拓扑数据路径——这在3+1架构中不仅是优化手段，更是NCC范式"物理拓扑即计算"的最佳例证。同一组物理连线，在AI模式下执行AllReduce通信，在信号处理模式下执行FFT计算——**这就是"网络即计算"的直接体现**。

## 五、研究内容（三年分解）

### Year 1（2026）：理论锚定 + 单FPGA原型

**目标**：在单颗Kria K26上跑通"3+1"架构的功能验证，发表核心理论论文。

#### 任务1.1 NCC-Edge架构设计与仿真（P0+P1，Q1-Q2）

**核心交付**：完成NCC-Edge-1的微架构规格、RTL设计（Chisel）、Verilator功能仿真。

具体工作：

**NPC-AR（AllReduce原语）设计**——这是整个MVP的灵魂模块。采用4节点Butterfly树架构，每个节点64bit数据位宽，内嵌定点加法器。开源起点：从Xilinx ACCL的AllReduce HLS实现中提取算法逻辑，用Chisel重写为参数化RTL生成器。LLM辅助：用Claude Code将ACCL的C++ HLS代码自动翻译为Chisel骨架，P1精修关键路径时序。

**CPC-M（矩阵乘原语）设计**——采用4×4脉动阵列（16 PE），INT8精度，128 MAC/cycle。开源起点：直接fork Berkeley Gemmini，裁剪到4×4 PE、去掉RISC-V RoCC接口、替换为SDIO-N Lite接口。这是Gemmini最常见的改造方式——Gemmini论文被引458次，大量follow-up工作都是从裁剪Gemmini开始的。

**CPC-S（FFT/并行前缀原语）设计**——采用Radix-2流水线FFT，1024点，复用Brent-Kung并行前缀网络。开源起点：ZipCPU的开源流水线FFT生成器（Verilog），参数化配置为1024点。**关键创新点**：CPC-S的蝴蝶数据路径与NPC-AR的蝴蝶拓扑物理共享——在RTL层面，这意味着同一组乘法器和加法器在AI模式下执行AllReduce的partial sum，在信号处理模式下执行FFT的butterfly运算。切换由SDI-Ctrl的模式寄存器控制。

**SDI-Ctrl（控制器）设计**——基于开源PicoRV32 RISC-V软核，加上拓扑查找表（BRAM）+ 模式切换状态机 + 4个性能计数器。P1用1-2周即可完成，大部分是已有IP的集成。

**SDIO-N Lite协议实现**——不做完整四层，只做L2（credit-based流控 + CRC-8）+ L3精简版（源路由 + 4种消息类型）+ L4精简版（TOPO_SWITCH + TOPO_QUERY两条命令）。从Constellation的channel/router模块裁剪。整个协议RTL目标<2000行Chisel。

**AI辅助重点**：

- Claude Code：Gemmini→CPC-M的接口适配代码自动生成
- RTLCoder：SDIO-N Lite的Flit编解码器、CRC-8、credit计数器
- Claude Opus：微架构规格文档撰写、各模块接口一致性审查

#### 任务1.2 cocotb验证套件与CI/CD（P2，Q1-Q3）

**核心交付**：覆盖3+1全部模块的自动化验证框架，Verilator仿真全部通过。

具体工作：

**每颗原语的独立验证**——NPC-AR的AllReduce正确性（参考值 vs 硬件输出逐bit对比）、CPC-M的矩阵乘正确性（与NumPy参考实现对比，INT8精度下误差为0）、CPC-S的FFT正确性（与scipy.fft对比，最大误差<1 LSB）。

**模块间集成验证**——CPC-M → NPC-AR流水线（先矩阵乘后AllReduce）、CPC-S ↔ NPC-AR路径复用（FFT-AllReduce同构验证）。

**CI/CD流水线**——GitHub Actions，每次代码commit自动触发Verilator仿真，生成覆盖率报告。

**AI辅助重点**：这是LLM最大价值区。P2的日常工作模式：

1. 写出测试场景描述（自然语言）
2. 喂给Claude，生成cocotb Python测试代码
3. 审查功能正确性，补充边界条件
4. 运行Verilator，失败日志喂给Claude分析root cause
5. Claude建议修复方案，P1确认后修改RTL

预期：200+测试用例中70%由LLM生成初稿，P2审查修改。

#### 任务1.3 Kria K26单板FPGA验证（P1+P4，Q3-Q4）

**核心交付**：在单颗Kria K26 SOM上运行3+1架构的完整功能demo。

此阶段的"4节点"是在同一颗FPGA内部用逻辑资源模拟的——4个CPC-M/CPC-S实例通过NPC-AR互连。物理上是一颗芯片，逻辑上是4节点NCC系统。这足以验证架构正确性和初步性能。

P4在此阶段的工作是为Year 2的双板设计做准备：选型、原理图框架设计、高速信号仿真。

#### 任务1.4 核心理论论文（P0，全年）

## 八、为什么这个MVP够用

回到最根本的问题：**NCC范式要证明什么？**

不是证明"我们能做出比Hailo更快的AI芯片"——那是产品竞争，不是范式验证。

要证明的是：“当计算被分解为Route + Transform、由物理网络拓扑原生承担通信操作时，同等硅资源上可以获得更高的有效算力和能效，并且一套硬件可以通过拓扑重构覆盖多个场景。”

NCC-Edge-1的"3+1"架构恰好是这个命题的**最小完备验证集**：

- NPC-AR证明了"通信可以被物理拓扑硬化"
- CPC-M + CPC-S证明了"计算可以被分解为极简原语"
- FFT-AllReduce同构路径证明了"物理拓扑即计算不是比喻，是字面含义"
- SDI拓扑热切换证明了"一套硬件、多个场景"
- 双FPGA板间互连证明了"可扩展"

当这块板卡摆在评审专家面前，你按一个按钮它在做视频AI，再按一个按钮它在做雷达波束成形，中间切换不到1微秒——**这比任何论文都有说服力**。

正如von Neumann在1945年没有一步做出大型计算机，而是先用ENIAC的17,468个真空管跑了一个氢弹数值模拟来证明"存储程序"概念可行——NCC-Edge-1就是NCC范式的ENIAC时刻。

核心架构问题：类ASIC vs 软件定义？  
答案：两者都不是，而是"软件定义 + 场景特化"的第三条路。

一、架构本质：原语芯粒 × SDI动态组合  
Copy❌ 错误理解A（纯类ASIC）：  
   超算专用板 / 智算专用板 / 信号专用板  
   → 三套硬件，三次流片，无法复用，经费爆炸

❌ 错误理解B（纯软件定义）：  
   9颗通用芯粒 + SDI随意连  
   → 太灵活，无法验证极致性能，验收场景模糊

✅ 正确方向：  
   9颗原语芯粒（每颗功能固定）

- SDI动态编排拓扑（软件定义连接方式）
- 三类场景配置文件（超算/智算/信号各一套）  
   = 同一套硬件，三种"拓扑模式"  
类比：芯粒是乐高积木（形状固定），SDI是拼法（软件定义），三种场景是三套拼图方案。

二、9颗芯粒的基本形态定义  
5种通信原语芯粒（NPC）  
芯粒	核心结构	规格（22nm）	面积	延迟  
NPC-AR AllReduce	Butterfly硬化树，归约节点内嵌加法器	64节点，FP32	~8mm²	45ns  
NPC-AA AlltoAll	全连接路由引擎，N×N交叉开关	32×32节点	~12mm²	80ns  
NPC-RS ReduceScatter	流水线分段归约，每段独立缓冲	64节点	~8mm²	60ns  
NPC-AG AllGather	径向多播引擎，1→N广播树	64节点	~6mm²	35ns  
NPC-BC Broadcast+Reduce	稀疏有向树，单点汇聚/发散	可配置深度	~5mm²	25ns  
关键设计原则：NPC芯粒只做数据搬运与归约，不做乘法，只有加法器+比较器。这是"简单节点"的工程边界。

4种算子原语芯粒（CPC）  
芯粒	核心结构	规格（22nm）	面积	吞吐  
CPC-M Map	脉动阵列（Systolic），8×8 PE	512 MAC/cycle，FP16	~15mm²	2 TOPS  
CPC-R Reduce	树形归约器，支持sum/max/min/avg	1024元素/cycle	~4mm²	—  
CPC-S Scan	Brent-Kung并行前缀树	1024元素/cycle	~6mm²	—  
CPC-T Transform	地址重排引擎 + 轻量查表	支持Softmax/GELU近似	~5mm²	—  
三、三类场景的拓扑配置方案  
场景1：超算（HPC科学计算）  
典型任务：CFD流体仿真、分子动力学、FFT

Copy拓扑配置：AllReduce-主导模式  
─────────────────────────────────  
NPC-AR（核心）── NPC-RS ── CPC-M  
     ↑                      ↓  
 Halo Exchange           局部矩乘  
 （边界数据同步）         （PDE迭代）

验证指标：  
✓ 1000×1000网格CFD：迭代通信时间 < 2μs  
✓ vs MPI+InfiniBand：通信延迟降低 10×  
✓ 扩展效率（64节点）：>0.92（GPU集群通常0.6-0.7）  
场景2：智算（AI推理）  
典型任务：L4实时推理（ResNet/YOLO）、多路视频分析

Copy拓扑配置：AlltoAll-主导模式（MoE场景）  
─────────────────────────────────────  
CPC-M（卷积主力）  
    ↕  （局部归约）  
CPC-R + CPC-T（BatchNorm + Softmax）  
    ↕  （跨设备专家路由）  
NPC-AA（AlltoAll，MoE专家分发）  
    ↕  （结果聚合）  
NPC-AR（AllReduce，梯度同步）

验证指标：  
✓ ResNet-50推理延迟：< 1ms（确定性）  
✓ 8路4K视频同步分析：功耗 < 30W  
✓ vs NVIDIA L4：有效算力利用率 > 85%（GPU通常20-30%）  
场景3：信号处理（雷达/通信）  
典型任务：OFDM FFT、CFAR目标检测、多天线波束成形

Copy拓扑配置：Scan+AllReduce融合模式  
──────────────────────────────────────
